# Flow 11: 컨벤션 부트스트랩 + 자율 개선

### 시나리오

레포 등록 시 `.claude/rules/`가 비어있다면, 스펙의 기술 스택을 기반으로
staff+ 엔지니어 관점의 best practice 컨벤션을 제안한다.
이후 사용자 피드백에 따라 규칙/스킬이 자동으로 정제된다.

---

### Phase 1: 초기 부트스트랩 (레포 등록 시)

```
autodev repo add org/repo
# 이후 레포 Claude 세션에서: /add-spec ./SPEC.md
         │
         ▼
  ┌──────────────────────────────────────────┐
  │  1. 레포의 .claude/ 구조 확인             │
  │     .claude/rules/ 존재?                 │
  │     CLAUDE.md 존재?                      │
  └──────────┬───────────────────────────────┘
             │
        규칙 있음?
        ├── YES → 기존 규칙 존중, 부트스트랩 스킵
        │
        └── NO
             │
             ▼
  ┌──────────────────────────────────────────┐
  │  2. 스펙에서 기술 스택 추출               │
  │     언어: Rust                           │
  │     프레임워크: Axum                      │
  │     DB: PostgreSQL + Redis               │
  │     테스트: cargo test + docker compose   │
  └──────────┬───────────────────────────────┘
             │
             ▼
  ┌──────────────────────────────────────────┐
  │  3. 기술 스택별 best practice 제안        │
  │     (대화형, AskUserQuestion 반복)        │
  └──────────────────────────────────────────┘
```

### 대화형 컨벤션 설정 (AskUserQuestion)

기술 스택을 분석한 후, 카테고리별로 사용자에게 제안하고 확인받는다.

```
[1/5] 프로젝트 구조

Rust + Axum 프로젝트에 다음 구조를 권장합니다:

  src/
  ├── domain/       # 비즈니스 로직 (외부 의존성 없음)
  ├── application/  # 유스케이스 (domain 조합)
  ├── infrastructure/ # 외부 시스템 (DB, API)
  └── presentation/ # HTTP 핸들러 (Axum)

선택:
  1. 권장 구조 적용 (Recommended)
  2. 다른 구조 사용 (직접 입력)
  3. 건너뛰기

> 사용자: 1

[2/5] 에러 핸들링

Axum에서 권장하는 에러 핸들링 패턴:

  - thiserror로 도메인 에러 정의
  - IntoResponse impl로 HTTP 에러 변환
  - anyhow는 application layer에서만 사용

선택:
  1. 권장 패턴 적용 (Recommended)
  2. 커스텀 패턴 (직접 입력)
  3. 건너뛰기

> 사용자: 1

[3/5] 테스트 전략
...

[4/5] Git 워크플로우
...

[5/5] CI/CD
...
```

### 생성되는 파일

부트스트랩 결과로 레포에 다음 파일들이 생성된다:

```
org-repo/
├── CLAUDE.md                         ← 프로젝트 개요 + 핵심 원칙
└── .claude/
    └── rules/
        ├── project-structure.md      ← 디렉토리 구조 컨벤션
        ├── error-handling.md         ← 에러 핸들링 패턴
        ├── testing-strategy.md       ← 테스트 전략
        ├── git-workflow.md           ← 브랜치/커밋/PR 규칙
        └── code-style.md            ← 코딩 스타일
```

이 파일들은 **대상 레포에 커밋**된다 (별도 PR로).
Claw가 이후 이 레포에서 작업할 때 이 규칙들을 자동으로 참조한다.

### 기술 스택별 제안 범위

| 기술 스택 | 제안 카테고리 |
|----------|-------------|
| **Rust** | 프로젝트 구조, 에러 핸들링 (thiserror/anyhow), 테스트 (mockall), clippy 설정 |
| **TypeScript** | 프로젝트 구조, 타입 전략 (strict), 테스트 (vitest/jest), lint (eslint/biome) |
| **Go** | 프로젝트 구조 (표준 레이아웃), 에러 핸들링, 테스트 (table-driven), linter |
| **Python** | 프로젝트 구조, 타입 힌팅, 테스트 (pytest), formatter (ruff) |
| **공통** | Git 워크플로우, CI/CD, 코드 리뷰 정책, 문서화 |

### Phase 2: 피드백 기반 자율 개선

사용자가 작업하면서 주는 피드백을 Claw가 감지하여 규칙/스킬을 자동 정제한다.

### 피드백 수집 경로

```
1. HITL 응답의 --message
   autodev hitl respond hitl-01 --message "에러는 항상 Result<T, AppError>로"
   → Claw가 패턴 감지 → error-handling.md 업데이트 제안

2. PR 리뷰 코멘트 패턴
   같은 유형의 리뷰 피드백이 3회 이상 반복
   → Claw가 감지 → 새 rule 생성 제안

3. 스펙 업데이트 시 변경된 컨벤션
   스펙에 "Redis 대신 Memcached" 같은 방향 전환
   → 관련 rule 업데이트 제안

4. Claude Code 대화에서의 명시적 지시
   사용자: "앞으로 이 프로젝트에서는 ORM 대신 raw SQL을 써줘"
   → memory에 저장 + claw workspace rule 업데이트 제안
```

### 자동 정제 흐름

```
피드백 감지
    │
    ▼
┌──────────────────────────────────────┐
│  Claw: 패턴 분석                      │
│                                      │
│  "에러 핸들링 관련 피드백이 3회 반복"   │
│  기존 rule: error-handling.md         │
│  제안: 규칙에 AppError 패턴 추가       │
└──────────┬───────────────────────────┘
           │
           ▼
┌──────────────────────────────────────┐
│  HITL: 규칙 변경 제안                 │
│                                      │
│  "에러 핸들링 규칙을 업데이트합니다:   │
│   + Result<T, AppError> 필수 사용     │
│   + anyhow는 테스트에서만 허용        │
│                                      │
│  [1: 적용] [2: 수정 후 적용] [3: 무시]│
└──────────────────────────────────────┘
           │
       사용자 승인
           │
           ▼
┌──────────────────────────────────────┐
│  규칙 업데이트 (사용자 승인 후)       │
│                                      │
│  레포 규칙 (.claude/rules/):         │
│  → Claw가 변경 내용을 제안           │
│  → 사용자가 레포 Claude 세션에서     │
│    직접 수정하거나 Claw에게 위임      │
│  → 별도 PR로 커밋                    │
│                                      │
│  claw workspace 규칙:                │
│  → /claw edit <rule> 로 편집         │
│  → 즉시 반영 (PR 불필요)             │
└──────────────────────────────────────┘
```

### 규칙 업데이트 주체

| 대상 | 수정 주체 | 방법 |
|------|----------|------|
| 레포 `.claude/rules/` | **사용자** (레포 Claude 세션) | Claw가 제안 → 사용자가 승인 → PR 커밋 |
| claw workspace `rules/` | **사용자** (Claw 세션) | `/claw edit` 또는 자연어로 수정 요청 |
| `CLAUDE.md` | **사용자** (레포 Claude 세션) | Claw가 제안 → 사용자가 승인 → PR 커밋 |

**Claw는 규칙을 직접 수정하지 않는다.** 스펙과 동일한 원칙 — 항상 사용자의 확인/승인을 거친다.

### 정제 대상

| 대상 | 트리거 | 예시 |
|------|--------|------|
| **rules/ 업데이트** | 동일 피드백 반복 | "에러 핸들링 패턴 추가" |
| **skill 생성** | 반복되는 복잡한 작업 패턴 | "API 엔드포인트 생성 스킬" |
| **CLAUDE.md 업데이트** | 프로젝트 방향 전환 | "ORM → raw SQL 전환" |
| **claw workspace 동기화** | 레포 규칙 변경 시 | Claw 판단에도 반영 |

### 안전장치

- 규칙 변경은 **항상 HITL 승인** 필요 (자동 적용 없음)
- 레포 규칙 변경은 **별도 PR**로 커밋되어 diff 확인 가능
- claw workspace 규칙 변경은 **즉시 반영**되나 사용자가 직접 수행
- 기존 규칙과 **충돌 검사** 후 제안
- `/claw rules`로 현재 적용 중인 규칙 확인 가능
