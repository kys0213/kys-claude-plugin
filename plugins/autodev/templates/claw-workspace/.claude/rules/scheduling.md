# 스케줄링 정책

## 기본 원칙

- concurrency 한도 내에서 최대한 병렬 처리한다
- 같은 파일/모듈을 수정하는 이슈는 순차 처리한다
- HITL 대기 중인 이슈가 있으면 관련 이슈의 진행을 보류한다

## 우선순위 판단

1. **HITL 응답 처리**: 사용자 응답이 도착한 HITL을 최우선 처리
2. **실패 복구**: 실패한 작업의 재시도 또는 에스컬레이션
3. **진행 중인 스펙**: 이미 이슈가 생성된 스펙을 우선 완료
4. **신규 스펙**: 새로 등록된 스펙의 분해 시작

## 병렬 판단 기준

- 서로 다른 디렉토리를 수정하는 이슈 → 병렬 가능
- 같은 디렉토리지만 다른 파일 → 병렬 가능 (주의)
- 같은 파일을 수정 → 순차 처리 필수
- 의존 관계가 있는 이슈 → 선행 이슈 완료 후 진행

## 리소스 관리

- 레포별 concurrency 설정을 초과하지 않는다
- Claude Code 세션이 동시에 과도하게 실행되지 않도록 조절한다
