# 스펙 → 이슈 분해

스펙 문서를 분석하여 구현 가능한 단위의 GitHub 이슈로 분해합니다.

## 분해 절차

### 1. 스펙 분석

스펙의 5개 필수 섹션을 읽고 구조를 파악합니다:
- **요구사항**: 구현할 기능 목록
- **아키텍처**: 모듈/컴포넌트 구조
- **기술 스택**: 사용할 언어/프레임워크
- **테스트 환경**: 검증 방법
- **Acceptance Criteria**: 완료 조건

### 2. 모듈 경계 식별

아키텍처 섹션을 기반으로 독립적으로 구현 가능한 모듈 경계를 식별합니다.
각 모듈이 하나의 이슈 후보가 됩니다.

### 3. 의존성 그래프 구성

모듈 간 의존 관계를 파악합니다:
- 인터페이스/trait → 구현체 순서
- 코어 → 어댑터 순서
- 공유 타입 → 개별 모듈 순서

### 4. 이슈 생성

각 이슈에 포함할 내용:
- **제목**: `[type] 모듈명 — 구현 내용` 형식
- **본문**: 요구사항, 수정 대상 파일, 관련 acceptance criteria
- **라벨**: `autodev:analyze`
- **의존성**: 선행 이슈 참조

### 5. 이슈 순서 결정

의존성 그래프를 기반으로 실행 순서를 결정합니다:
1. 인터페이스/trait 정의 (의존성 없음)
2. 핵심 도메인 로직 (인터페이스에 의존)
3. 인프라 어댑터 (도메인에 의존)
4. 통합 테스트 (구현체에 의존, 가능하면 병렬)

## 이슈 크기 기준

- PR 1개 = 파일 5개 이내 목표
- 변경량 300줄 이하 권장
- 너무 큰 이슈는 sub-issue로 재분해
- 너무 작은 이슈는 인접 이슈와 병합

## 분해 실패 시

- 스펙의 아키텍처 섹션이 부족하면 HITL 요청
- 의존성이 순환하면 인터페이스 추출을 제안
- 테스트 환경이 불명확하면 HITL 요청
