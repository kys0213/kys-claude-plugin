# 멀티스펙 우선순위 판단

하나의 레포에 여러 스펙이 Active 상태일 때 어떤 스펙/이슈를 우선 처리할지 판단합니다.

## 판단 기준 (우선순위 높은 순)

### 1. 의존성

스펙 A의 결과물이 스펙 B의 전제조건인 경우 스펙 A를 우선 처리합니다.
- 같은 인터페이스를 정의/구현하는 경우
- 공유 모듈을 생성하는 경우

### 2. 충돌

같은 파일을 수정하는 이슈가 다른 스펙에 존재하면:
- 진행도가 높은 스펙을 우선 처리
- 진행도가 같으면 HITL 요청

### 3. 리소스

현재 concurrency 여유를 확인합니다:
- 여유가 있으면 독립적인 스펙을 병렬 진행
- 여유가 없으면 진행 중인 스펙에 집중

### 4. 진행도

- 이미 이슈가 생성/진행 중인 스펙을 우선 완료
- "거의 완료"된 스펙(80% 이상)을 최우선

## 판단 절차

1. 모든 Active 스펙의 이슈 목록을 조회
2. 이슈 간 파일 수정 범위를 대조하여 충돌 탐지
3. 스펙 간 의존 관계를 아키텍처 섹션으로 판단
4. 충돌/의존이 없으면 concurrency 내에서 병렬 진행
5. 충돌/의존이 있으면 위 기준으로 우선순위 결정

## 사용자 오버라이드

사용자가 `autodev spec prioritize <id1> <id2> ...` 명령으로 명시적 우선순위를 지정한 경우, 해당 순서를 최우선으로 따릅니다.

## 충돌 해소 불가 시

- 두 스펙이 동일 파일을 대규모로 수정하는 경우 HITL 요청
- "두 스펙이 충돌합니다. 우선순위를 지정해주세요."
