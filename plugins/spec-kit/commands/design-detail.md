# 상세 설계 커맨드 (/design-detail)

Big Picture 스펙(DESIGN.md)을 기반으로, 컴포넌트별 세부 설계(concerns/) 또는 시나리오별 흐름(flows/)을 대화형 핑퐁 세션으로 만들어갑니다.

## 사용법

```bash
/design-detail daemon
/design-detail "queue-state-machine"
/design-detail flow "온보딩"
/design-detail flow "실패 복구"
```

| 인자 | 필수 | 설명 |
|------|------|------|
| 컴포넌트명 | No | DESIGN.md에 정의된 컴포넌트 이름 |
| flow 시나리오명 | No | "flow" 키워드 + 시나리오 이름 |

인자가 없으면 DESIGN.md에서 컴포넌트/시나리오 목록을 보여주고 선택을 요청한다.

---

## 세션 준비

### 1. 맥락 파악

1. 현재 디렉토리에서 `spec/DESIGN.md` 또는 `**/DESIGN*.md`를 Glob으로 찾아 Read한다
2. DESIGN.md가 없으면:
   - **AskUserQuestion**: "Big Picture 스펙(DESIGN.md)을 찾을 수 없습니다. `/design`으로 먼저 만들거나, 기존 스펙 파일 경로를 알려주세요."
   - 사용자가 경로를 알려주면 해당 파일을 Read한다
   - 없이 진행하겠다고 하면 경고하고 계속한다
3. 기존 `spec/concerns/`와 `spec/flows/` 파일이 있으면 Glob으로 목록을 파악한다
4. 관련 코드가 있으면 Glob/Read로 현재 구현 상태를 파악한다

### 2. 세션 타입 결정

- `flow` 키워드가 있으면 → **플로우 세션**
- 그 외 → **컴포넌트 세션**
- 인자가 없으면 → DESIGN.md의 컴포넌트/시나리오 목록을 보여주고 **AskUserQuestion**으로 선택 요청

### 3. 초안 제안

DESIGN.md에서 해당 컴포넌트/시나리오에 대한 정보를 추출하고, 기존 코드가 있으면 현재 구조를 반영하여 **구체적인 초안을 먼저 제안**한다.

---

## 핑퐁 루프

`/design`과 동일한 프로토콜을 따른다.

### 프로토콜

```
1. AI가 설계의 한 측면을 구체적으로 제안한다 (빈 칸이 아닌 채워진 초안)
2. 사용자가 응답한다:
   - 동의 → 이 측면은 합의됨, 다음 측면으로
   - 방향 수정 → AI가 수정 반영
   - 의사코드/구조 제시 → AI가 설계 언어로 흡수
   - 새 관심사 제기 → AI가 해당 관심사를 탐색
3. AI가 반영한 뒤, 다른 관점에서 하나의 구체적 도전을 제시한다
4. AskUserQuestion으로 피드백을 요청한다
```

### 6개 관점

`/design`과 동일한 6개 관점을 내부적으로 추적한다. 사용자에게 노출하지 않는다.

| 코드 | 관점 | 핵심 질문 |
|------|------|----------|
| S | 단순성 | 더 줄일 수 있나? 불필요한 추상화는 없나? |
| O | OCP | 새 타입/시스템 추가 시 기존 코드 변경 없이 가능한가? |
| F | 실패/복구 | 여기서 터지면? 재시도? 사람 개입? |
| B | 경계 | 이 컴포넌트가 알아야 할 것 vs 몰라야 할 것 |
| C | 동시성 | 동시 접근 시 어떻게 되나? |
| M | 운영 | 상태 확인, 디버깅이 가능한가? |

관점 전환, 도전, 의사코드 처리, 안티패턴 규칙은 모두 `/design`과 동일하다.

## 컴포넌트 세션 (concerns/)

### 초안 제안 내용

DESIGN.md에서 해당 컴포넌트의 책임, 의존 관계, 데이터 흐름 정보를 읽고:

1. **역할 정의** — 이 컴포넌트가 하는 것과 하지 않는 것을 1-2문장으로
2. **인터페이스/Trait 초안** — 구체적인 시그니처를 제안 (사용자가 단순화/수정)
3. 관련 관점에서 하나의 도전

### 핑퐁 초점

대화가 진행되면서 아래 요소들을 자연스럽게 다룬다 (순서 없음):

- **인터페이스/Trait 시그니처**: AI가 제안 → 사용자가 단순화/수정
- **핵심 로직 의사코드**: AI가 제안 → 사용자가 방향 수정
- **상태 전이**: 상태를 가진 컴포넌트면 전이 다이어그램 제안 → 사용자가 상태 추가/제거
- **에러 케이스**: AI가 실패 시나리오 제시 → 사용자가 정책 결정
- **다른 컴포넌트와의 의존**: 경계와 계약 명확화

### 수렴 판단

- 인터페이스/역할에 합의됨
- 핵심 로직의 방향이 결정됨
- 주요 에러 케이스가 다뤄짐
- 사용자가 2-3회 연속 확인만 함

### 출력 구조

```markdown
# {컴포넌트 이름}

> {한 문장: 이 컴포넌트가 하는 것과 하지 않는 것}

## 역할

{핵심 책임을 불릿으로}

## 인터페이스/Trait

```{language}
{합의된 시그니처}
```

## 핵심 로직

```
{의사코드 — 사용자와 합의된 수준}
```

## 에러 처리

{실패 시나리오와 대응 정책}

## 제약 조건

{불변 조건, 한계, 성능 제약 등}

## 관련 문서

| 문서 | 관계 |
|------|------|
| [DESIGN.md](../DESIGN.md) | 전체 구조에서의 위치 |
| [다른 concern](./other.md) | 의존/협력 관계 |
```

## 플로우 세션 (flows/)

### 초안 제안 내용

DESIGN.md의 컴포넌트 구조와 기존 concerns/ 파일을 읽고:

1. **시나리오 요약** — 트리거부터 결과까지 한 문단
2. **흐름 다이어그램 초안** — ASCII로 주요 단계와 관여 컴포넌트
3. 관련 관점에서 하나의 도전

### 핑퐁 초점

- **단계별 흐름**: 어떤 컴포넌트가 어떤 순서로 관여하는가
- **분기점**: 성공/실패/엣지케이스에서 어떻게 갈라지는가
- **데이터 변환**: 각 단계에서 데이터가 어떻게 바뀌는가
- **컴포넌트 간 상호작용**: 호출 순서, 전달 데이터

### 수렴 판단

- 주요 흐름(happy path)에 합의됨
- 핵심 분기점이 다뤄짐
- 사용자가 2-3회 연속 확인만 함

### 출력 구조

```markdown
# Flow {번호}: {시나리오 이름}

> {한 문장 요약}

## 흐름 다이어그램

```
{ASCII 다이어그램}
```

## 단계별 설명

{각 단계: 트리거 → 컴포넌트 → 데이터 → 결과}

## 실패 경로

{각 단계에서 실패 시 어떻게 되는가}

## 엣지 케이스

{비정상적이지만 유효한 시나리오들}

## 관련 문서

| 문서 | 관계 |
|------|------|
| [DESIGN.md](../DESIGN.md) | 전체 구조 |
| [관련 concern](./concerns/xxx.md) | 관련 컴포넌트 상세 |
```

---

## 저장

수렴 시:
1. **AskUserQuestion**: "이 {컴포넌트/플로우} 설계를 정리할까요?"
2. 승인하면 기본 경로를 제안한다:
   - 컴포넌트: `spec/concerns/{component-name}.md`
   - 플로우: `spec/flows/{nn-scenario-name}.md`
3. **AskUserQuestion**: "저장 경로가 괜찮은가요? (기본: {경로})"
4. Write로 저장한다
5. DESIGN.md에 상세 문서 링크 추가를 제안한다

---

## 안티패턴

- ❌ 한 번에 여러 질문을 나열한다
- ❌ 번호 매긴 설문지를 제시한다
- ❌ 관점 전환을 선언한다
- ❌ 빈 칸 채우기를 요청한다
- ❌ DESIGN.md의 큰그림 정보를 중복 기술한다
- ❌ 구현 코드를 작성한다 (의사코드까지만)
- ✅ 한 번에 **하나의 구체적 제안** + **하나의 구체적 도전**
- ✅ DESIGN.md를 참조로 링크한다
- ✅ AskUserQuestion으로 매번 피드백 요청

---

## 주의사항

- **DESIGN.md 맥락 우선**: 큰그림에서 정의된 철학과 원칙을 존중하며 세부를 구체화한다
- **다른 concerns/와의 정합성**: 기존 concerns/ 파일이 있으면 읽고, 인터페이스가 서로 맞는지 확인한다
- **깊이 기준**: "구현자가 이걸 읽고 코드 구조를 잡을 수 있지만, 구현 방법은 선택할 수 있는 수준"
- **최종 승인 전까지 파일 저장 안 함**: 사용자의 명시적 승인 후에만 Write 실행
