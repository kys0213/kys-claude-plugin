# 설계 퍼실리테이션 커맨드 (/design)

Use case와 컴포넌트 아키텍처 관점의 스펙을 대화형으로 구조화합니다.
구체적인 구현 세부사항(함수 시그니처, DB 스키마 등)은 다루지 않습니다.
최종 컨펌이 떨어질 때까지 AskUserQuestion을 활용하여 반복적으로 정제합니다.

## 사용법

```bash
/design
/design "사용자 인증 시스템"
/design "결제 파이프라인 리팩토링"
```

| 인자 | 필수 | 설명 |
|------|------|------|
| 주제 | No | 설계 대상 주제 (미지정 시 대화로 도출) |

## 작업 프로세스

이 커맨드는 에이전트를 사용하지 않고, 직접 대화형으로 진행합니다.

### Phase 1: 목표 & 범위 정의

주제가 주어졌으면 이를 기반으로, 없으면 아래 질문부터 시작합니다:

**AskUserQuestion**:
1. "이 시스템/기능이 해결하려는 핵심 문제는 무엇인가요?"
2. "주요 사용자(Actor)는 누구이고, 각각 어떤 목적을 가지나요?"
3. "시스템의 경계는 어디까지인가요? (포함/제외 범위)"

Phase 1 완료 시 정리하여 확인:
```markdown
## 목표 & 범위
- **핵심 문제**: ...
- **Actor**: ...
- **범위**: 포함 [...] / 제외 [...]
```

**AskUserQuestion**: "위 정리가 맞나요? 수정할 부분이 있으면 말씀해주세요."

### Phase 2: Use Case 도출

Actor별 주요 사용 시나리오를 도출합니다:

**AskUserQuestion**:
1. "[Actor A]가 이 시스템으로 하는 가장 중요한 3가지 행위는?"
2. "각 행위에서 예외적인 상황(실패, 취소, 재시도)은 어떤 것들이 있나요?"
3. "Actor 간 상호작용이 있나요? (예: A의 행위 결과가 B에게 영향)"

Phase 2 완료 시 정리:
```markdown
## Use Cases

### UC-1: [Use Case 이름]
- **Actor**: ...
- **Trigger**: ...
- **주요 흐름**: 1. ... → 2. ... → 3. ...
- **예외 흐름**: ...
- **결과**: ...
```

**AskUserQuestion**: "도출된 Use Case 목록을 확인해주세요. 빠진 시나리오가 있나요?"

### Phase 3: 컴포넌트 설계

Use Case를 실현할 컴포넌트 구조를 설계합니다:

기존 코드베이스가 있으면 Glob/Read로 현재 구조를 파악한 뒤, 변경/추가가 필요한 부분을 식별합니다.

정리:
```markdown
## 컴포넌트 설계

### 컴포넌트 목록
| 컴포넌트 | 책임 | 의존 대상 |
|----------|------|----------|

### 데이터 흐름
[UC별 데이터가 컴포넌트를 통과하는 흐름]

UC-1: Actor → [컴포넌트A] → [컴포넌트B] → [외부시스템] → 결과

### 외부 의존성
| 외부 시스템 | 역할 | 인터페이스 |
|------------|------|----------|
```

**AskUserQuestion**: "컴포넌트 구조와 데이터 흐름을 확인해주세요. 책임 분리나 의존성 방향에 대해 의견이 있으신가요?"

### Phase 4: 최종 확인 & 저장

Phase 1~3의 결과를 하나의 스펙 문서로 통합합니다.

```markdown
# [주제] 설계 스펙

## 1. 목표 & 범위
[Phase 1 결과]

## 2. Use Cases
[Phase 2 결과]

## 3. 컴포넌트 설계
[Phase 3 결과]

## 4. 미결정 사항
[대화 중 확인이 필요하다고 표시된 항목]

## 5. 다음 단계
- [ ] 상세 인터페이스 정의
- [ ] 테스트 시나리오 작성
- [ ] 구현 계획 수립
```

**AskUserQuestion**: "최종 스펙 초안을 확인해주세요. 승인하시면 파일로 저장합니다. 저장 경로를 지정해주세요 (예: docs/auth-design.md)"

사용자가 경로를 지정하면 Write로 저장합니다.
수정 요청이 오면 해당 Phase로 돌아가서 반복합니다.

## 다루지 않는 것

- 함수/메서드 시그니처
- DB 스키마, 테이블 설계
- API endpoint 상세 (request/response body)
- 구현 언어나 프레임워크 선택
- 코드 수준의 설계 패턴 (이것들은 구현 단계에서 결정)

## 주의사항

- **매 Phase마다 사용자 확인**: 다음 Phase로 넘어가기 전에 반드시 AskUserQuestion으로 확인
- **수정 요청 시 돌아가기**: 사용자가 이전 Phase의 내용을 수정하고 싶으면 해당 Phase로 돌아감
- **기존 코드 참조**: 코드베이스가 있으면 현재 구조를 파악하여 설계에 반영하되, 설계가 기존 구조에 종속되지 않도록 주의
- **최종 컨펌 전까지 저장 안 함**: 사용자의 명시적 승인 후에만 파일 저장
