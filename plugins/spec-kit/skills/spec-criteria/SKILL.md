# Spec Quality Criteria

스펙 문서의 완성도를 평가하는 4가지 관점과 각 관점의 세부 체크리스트입니다.

## 관점 A: Big Picture (전체 구조)

시스템의 전체 그림이 잡히는가.

| # | 체크 항목 | 미충족 시 문제 |
|---|----------|---------------|
| A1 | 시스템 목적과 해결하려는 문제가 명시됨 | 구현 방향이 모호해짐 |
| A2 | 주요 Actor(사용자/외부시스템)가 식별됨 | Use case 도출 불가 |
| A3 | 핵심 컴포넌트가 나열되고 각 책임이 서술됨 | 모듈 분리 기준이 없음 |
| A4 | 컴포넌트 간 의존 관계/방향이 표현됨 | 순환 의존, 결합도 판단 불가 |
| A5 | 주요 데이터 흐름이 추적 가능함 | 통합 테스트 설계 불가 |
| A6 | 외부 시스템과의 경계가 명확함 | 책임 범위 혼란 |
| A7 | 포함/제외 범위(Scope)가 정의됨 | 범위 확장(scope creep) 발생 |

## 관점 B: Detail (상세 정의)

각 요구사항이 구현 가능한 수준으로 구체적인가.

| # | 체크 항목 | 미충족 시 문제 |
|---|----------|---------------|
| B1 | 각 요구사항의 입력이 정의됨 (타입, 범위, 필수 여부) | 입력 검증 기준 부재 |
| B2 | 각 요구사항의 출력이 정의됨 (형태, 내용) | 반환값 구현 기준 부재 |
| B3 | 예외 상황이 열거됨 (에러 종류, 실패 시 동작) | 에러 처리 누락 |
| B4 | 경계값/제약 조건이 명시됨 (최대 길이, 허용 범위 등) | 엣지 케이스 미처리 |
| B5 | 용어가 모호하지 않게 정의됨 ("빠르게", "적절히" 등 배제) | 구현자마다 해석이 다름 |
| B6 | 상태 전이가 있으면 전이 조건이 명시됨 | 잘못된 상태 전이 발생 |
| B7 | 비기능 요구사항이 정량적으로 표현됨 (응답시간 < 200ms) | 성능/보안 기준 부재 |

## 관점 C: Verification (검증 가능성)

각 요구사항을 어떻게 검증할 수 있는가.

| # | 체크 항목 | 미충족 시 문제 |
|---|----------|---------------|
| C1 | 수용 기준(Acceptance Criteria)이 존재함 | 완료 판단 기준 부재 |
| C2 | 성공/실패 판단 기준이 정량적임 | 주관적 판단에 의존 |
| C3 | 테스트에 필요한 환경 요건이 명시됨 | 테스트 환경 구축 지연 |
| C4 | 필요한 외부 서비스/Mock이 정의됨 | 통합 테스트 불가 |
| C5 | 테스트 데이터 요건이 명시됨 | 테스트 커버리지 부족 |
| C6 | 성능 테스트 기준이 있음 (부하, 동시 접속 등) | 성능 검증 불가 |

## 관점 D: Consistency (일관성)

문서 전반이 모순 없이 일관적인가.

| # | 체크 항목 | 미충족 시 문제 |
|---|----------|---------------|
| D1 | 동일 개념에 동일 용어 사용 | 혼란, 오구현 |
| D2 | 요구사항 간 모순 없음 | 양립 불가능한 요구사항 구현 시도 |
| D3 | 인터페이스 정의가 양쪽에서 일치 | 통합 시 불일치 |
| D4 | 참조된 컴포넌트/요구사항이 모두 정의되어 있음 | 누락된 의존성 |
| D5 | 우선순위/중요도가 일관된 기준으로 부여됨 | 구현 순서 혼란 |

## 점수 산정 기준

각 관점의 체크 항목 충족률로 점수를 산정합니다:

```
점수 = (충족 항목 수 / 전체 항목 수) × 100
```

단, 항목별 가중치를 적용합니다:
- 충족: 1.0
- 부분 충족 (일부만 서술): 0.5
- 미충족: 0.0

### 등급

| 등급 | 점수 | 의미 |
|------|------|------|
| A | 90-100 | 구현 즉시 가능 |
| B | 70-89 | 소수 보완 후 구현 가능 |
| C | 50-69 | 상당한 보완 필요 |
| D | 0-49 | 근본적 재작성 필요 |
