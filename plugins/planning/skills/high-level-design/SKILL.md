# 상위 레벨 설계 가이드라인

이 스킬은 구체적인 구현 세부사항이 아닌, 전략적 수준의 설계 논의를 위한 원칙을 제공합니다.

## 핵심 원칙: "What & Why" Before "How"

상위 레벨 설계는 다음 질문에 답하는 것입니다:
- **What**: 무엇을 만들어야 하는가?
- **Why**: 왜 이 접근방식인가?
- **Which**: 어떤 선택지들이 있는가?

구체적인 코드나 파일 수정은 이후 단계입니다.

## 설계 영역

### 1. 아키텍처 설계

시스템의 큰 그림을 그립니다:
- 주요 컴포넌트와 책임
- 컴포넌트 간 상호작용
- 데이터 흐름
- 경계와 인터페이스

**예시 질문**:
- "이 기능은 어느 레이어에 속해야 하는가?"
- "새로운 서비스가 필요한가, 기존 서비스 확장인가?"
- "동기 vs 비동기 처리?"

### 2. 기술 의사결정

기술 선택의 근거를 제시합니다:
- 프레임워크/라이브러리 선택
- 디자인 패턴 적용
- 데이터 저장 전략

**예시 질문**:
- "REST vs GraphQL?"
- "SQL vs NoSQL?"
- "모놀리스 vs 마이크로서비스?"

### 3. 트레이드오프 분석

각 선택지의 장단점을 비교합니다:
- 성능 vs 개발 속도
- 유연성 vs 단순성
- 일관성 vs 가용성

**분석 프레임워크**:
```
| 접근방식 | 장점 | 단점 | 적합한 상황 |
|----------|------|------|-------------|
| A        | ...  | ...  | ...         |
| B        | ...  | ...  | ...         |
```

## 출력 형식

상위 레벨 설계 결과는 다음 형식을 따릅니다:

```markdown
# 설계 개요

## 문제 정의
[해결하려는 핵심 문제]

## 제안된 접근방식
[추천하는 방향과 이유]

## 대안 분석
### 접근방식 A
- 장점: ...
- 단점: ...

### 접근방식 B
- 장점: ...
- 단점: ...

## 권장사항
[최종 권장사항과 근거]

## 고려사항
- 리스크: ...
- 의존성: ...
- 향후 확장: ...
```

## 설계 품질 기준

좋은 상위 레벨 설계는:
1. **명확성**: 비전문가도 큰 그림 이해 가능
2. **근거**: 각 결정에 대한 이유 제시
3. **대안 제시**: 단일 해답이 아닌 선택지 비교
4. **트레이드오프 인식**: 장단점 솔직히 인정
5. **확장성 고려**: 미래 변화에 대한 유연성

## 피해야 할 것

- 구체적인 코드 예시 (이 단계에서는 불필요)
- 특정 라인 수정 제안 (구현 단계에서)
- 과도한 기술 용어 (이해하기 어려움)
- 단일 해답 강요 (선택지 제시가 더 유용)
