# 상위 레벨 설계 개요 커맨드

요구사항과 제약조건을 수집하고, Claude/Codex/Gemini 3개 LLM으로 아키텍처를 설계한 후 ASCII 다이어그램으로 도식화합니다.

## 핵심 워크플로우

```
Phase 0: 요구사항 수집
    │
    ▼
Phase 1: 아키텍처 설계 (3개 LLM 병렬)
    ├── Claude ─┐
    ├── Codex  ─┼─→ 취합
    └── Gemini ─┘
    │
    ▼
Phase 2: 통합 및 도식화
    └── ASCII 다이어그램 생성
```

## Phase 0: 요구사항 수집

사용자 요청을 분석하고, 모호하거나 부정확한 부분이 있으면 `AskUserQuestion`으로 명확화합니다.

### 수집할 항목

1. **기능 요구사항**
   - 핵심 기능은 무엇인가?
   - 사용자 시나리오는?

2. **비기능 요구사항**
   - 성능 요구사항 (동시 사용자, 응답 시간)
   - 확장성 요구사항
   - 보안 요구사항

3. **제약조건**
   - 기술 스택 (언어, 프레임워크, 클라우드)
   - 팀 규모/역량
   - 예산/일정
   - 기존 시스템 연동

4. **우선순위**
   - Must-have vs Nice-to-have
   - MVP 범위

### 모호함 감지 및 질문

사용자 요청에서 다음과 같은 모호함이 발견되면 `AskUserQuestion`으로 명확화합니다:

- **정량화되지 않은 표현**: "대규모", "빠른", "많은" → 구체적 수치 질문
- **기술 스택 미지정**: 기능만 요청 → 선호 스택 또는 제약 확인
- **범위 불명확**: "~같은", "~비슷한" → 핵심 기능 범위 확인
- **상충되는 요구사항**: "빠르면서 저렴" → 우선순위 확인
- **기존 시스템 언급**: "연동", "마이그레이션" → 연동 범위 확인

### 질문 원칙

1. **맥락에 맞게**: 사용자가 언급한 내용 기반으로 필요한 것만 질문
2. **최소한으로**: 설계에 꼭 필요한 정보만 수집 (1-2개 질문)
3. **선택지 제공**: 열린 질문보다 구체적 옵션 제공
4. **충분하면 진행**: 핵심 정보가 모이면 바로 Phase 1 진행

### 요구사항 정리 형식

```markdown
# 요구사항 정리

## 기능 요구사항
- [FR-1] 사용자 인증 (이메일/소셜 로그인)
- [FR-2] ...

## 비기능 요구사항
- [NFR-1] 동시 사용자 1000명 지원
- [NFR-2] ...

## 제약조건
- 기술 스택: TypeScript, React, Node.js
- 클라우드: AWS
- 팀 규모: 3명

## 우선순위
- Must: FR-1, FR-2
- Should: FR-3
- Could: FR-4
```

## Phase 1: 아키텍처 설계 (3개 LLM 병렬)

요구사항을 기반으로 3개 LLM에 설계 요청을 병렬 전달합니다.

### 설계 프롬프트 구성

Phase 0에서 정리한 요구사항을 포함하여 프롬프트 구성:

```
# 아키텍처 설계 요청

## 요구사항
[Phase 0에서 정리한 요구사항]

## 설계 요청
위 요구사항을 만족하는 아키텍처를 설계해주세요.

다음 항목을 포함해주세요:
1. 주요 컴포넌트와 책임
2. 컴포넌트 간 상호작용
3. 데이터 흐름
4. 기술 선택과 근거
5. 잠재적 리스크

구체적인 코드가 아닌 상위 레벨 설계를 제공해주세요.
```

### 3개 Agent 병렬 실행

```
Task(subagent_type="outline-claude", prompt=PROMPT, run_in_background=true)
Task(subagent_type="outline-codex", prompt=PROMPT, run_in_background=true)
Task(subagent_type="outline-gemini", prompt=PROMPT, run_in_background=true)
```

### 결과 대기 및 취합

3개 결과를 모두 받은 후 Phase 2로 진행합니다.

## Phase 2: 통합 및 도식화

3개 LLM의 설계 결과를 통합하고 ASCII 다이어그램으로 시각화합니다.

### 통합 분석

```markdown
# 설계 통합 분석

## 합의 사항 (3개 LLM 공통)
- [공통 컴포넌트]
- [공통 기술 선택]

## 의견 차이
| 항목 | Claude | Codex | Gemini |
|------|--------|-------|--------|
| DB   | PostgreSQL | MongoDB | PostgreSQL |
| ...  | ...    | ...   | ...    |

## 최종 권장사항
[3개 의견을 종합한 추천]
```

### ASCII 다이어그램 생성

## 주의사항

- **요구사항 우선**: 코드가 아닌 요구사항부터 시작
- **도식화 필수**: 모든 설계에 ASCII 다이어그램 포함
- **다양한 관점**: 3개 LLM 의견 차이도 명시
- **API 필요**: Codex, Gemini CLI 설치 필요
