# Spec Validation Criteria

autopilot 자율주행을 시작하기 전에 스펙 품질을 평가하는 3가지 구조적 관점과 판정 기준입니다.

## 관점 A: Flow Completeness (엣지케이스 커버리지)

스펙이 모든 실행 경로를 커버하여 구현자가 즉시 작업 가능한 수준인지 평가합니다.

| # | 체크 항목 | 미충족 시 위험 |
|---|----------|---------------|
| FC1 | 각 주요 흐름의 happy path가 입력→처리→출력으로 정의됨 | 구현자가 정상 흐름을 추측해야 함 |
| FC2 | 각 주요 흐름의 실패/예외 경로가 정의됨 | 에러 처리가 누락되거나 임의 구현됨 |
| FC3 | 프론트엔드/백엔드 각각의 구현 범위가 명확함 | 책임 경계 모호로 중복/누락 발생 |
| FC4 | 시스템 간 상호작용 실패 시나리오가 커버됨 | 네트워크 장애, 타임아웃 등 미처리 |
| FC5 | 보상 트랜잭션(롤백, 재시도, 폴백)이 명시됨 | 부분 실패 시 데이터 불일치 |

### spec-kit 관점 매핑

Flow Completeness는 spec-kit의 **A(구조) + B(상세)** 교차 영역에 해당합니다:

| spec-kit 항목 | Flow Completeness 대응 |
|--------------|----------------------|
| A3 (컴포넌트 책임 서술) | FC1, FC3 — 각 컴포넌트가 담당하는 흐름의 범위 |
| A5 (데이터 흐름 추적) | FC1 — happy path의 데이터 흐름 |
| B3 (예외 상황 열거) | FC2, FC4 — 실패/예외 경로 |
| B6 (상태 전이 조건) | FC5 — 실패 시 보상 트랜잭션 |

## 관점 B: Boundary Clarity (컴포넌트 경계 정의)

시스템 구성 요소 간 경계가 구현에 필요한 수준으로 명확히 정의되어 있는지 평가합니다.

| # | 체크 항목 | 미충족 시 위험 |
|---|----------|---------------|
| BC1 | 시스템 수준 컴포넌트 경계가 다이어그램 또는 명시적 서술로 표현됨 | 모듈 분리 기준 부재 |
| BC2 | 각 컴포넌트의 입력/출력/부수효과가 정의됨 | 인터페이스 불일치 |
| BC3 | 의존성 방향이 명확하고 순환 의존이 없음 | 빌드/배포 순서 혼란 |
| BC4 | 외부 시스템 인터페이스(API 계약)가 정의됨 | 통합 시 불일치 |

### spec-kit 관점 매핑

Boundary Clarity는 spec-kit의 **A(구조) + D(일관성)** 영역에 해당합니다:

| spec-kit 항목 | Boundary Clarity 대응 |
|--------------|---------------------|
| A4 (컴포넌트 간 의존 관계) | BC3 — 의존성 방향 명확성 |
| A6 (외부 시스템 경계) | BC1, BC4 — 시스템 경계와 외부 인터페이스 |
| D3 (인터페이스 정의 일치) | BC2, BC4 — 컴포넌트 간 계약 일치 |
| D4 (참조 컴포넌트 정의 여부) | BC1 — 언급된 컴포넌트의 실제 정의 존재 |

## 관점 C: OCP Structurability (확장성 설계)

스펙이 OCP(개방-폐쇄 원칙)를 따르는 구현으로 자연스럽게 이어질 수 있는 구조인지 평가합니다.

| # | 체크 항목 | 미충족 시 위험 |
|---|----------|---------------|
| OCP1 | 새 타입/메서드 추가 시 기존 코드 수정 없이 확장 가능한 구조 | 기능 추가마다 광범위한 수정 필요 |
| OCP2 | 상태 전이가 다형성/전략 패턴으로 표현 가능 | 분기문 산재, 유지보수 비용 증가 |
| OCP3 | 비즈니스 규칙 변경 지점이 식별되고 영향 범위가 예측 가능 | 변경 시 사이드이펙트 예측 불가 |
| OCP4 | 추상과 구체 구현의 분리가 설계 수준에서 반영됨 | 구체 구현에 직접 의존하는 코드 양산 |

### spec-kit 관점 매핑

OCP Structurability는 spec-kit의 기존 4관점(A-D)으로 **직접 커버되지 않는 신규 축**입니다:

- spec-kit은 "스펙 문서의 완성도"를 평가하지만, OCP Structurability는 "스펙이 좋은 코드 구조로 이어지는가"를 평가합니다.
- opus 모델의 설계 판단 능력을 활용하여, 스펙 단계에서 구현 코드의 확장성을 사전 평가합니다.

## 판정 기준

### 개별 관점 판정

| 판정 | 기준 |
|------|------|
| **PASS** | 해당 관점의 체크 항목이 80% 이상 충족됨 |
| **WARN** | 해당 관점의 체크 항목이 50% 이상 80% 미만 충족됨 |
| **FAIL** | 해당 관점의 체크 항목이 50% 미만 충족됨 |

### 충족 판정 세부 기준

각 체크 항목은 다음 중 하나로 판정합니다:
- **충족 (1.0)**: 해당 항목이 스펙에 명시적으로 정의되어 있음
- **부분 충족 (0.5)**: 일부만 서술되어 있거나 암시적으로 유추 가능
- **미충족 (0.0)**: 전혀 언급되지 않거나 판단 불가

충족률 = (각 항목 점수의 합 / 전체 항목 수) x 100

### Overall 판정

| 조건 | overall |
|------|---------|
| 3가지 관점 모두 PASS | PASS |
| FAIL 없고 WARN 존재 | WARN |
| FAIL이 하나라도 존재 | FAIL |

## FAIL 시 스펙 개선 가이드

### Flow Completeness FAIL 개선

1. 각 주요 기능의 시퀀스 다이어그램 또는 흐름도를 작성
2. 각 흐름에 대해 "정상 → 실패 → 복구" 3가지 경로를 명시
3. 시스템 간 통신 실패 시 동작을 구체적으로 기술
4. 재시도 정책, 타임아웃 값, 폴백 전략을 수치로 명시

### Boundary Clarity FAIL 개선

1. 시스템 아키텍처 다이어그램 추가 (컴포넌트 + 의존성 방향)
2. 각 컴포넌트의 책임을 한 문장으로 정의
3. 컴포넌트 간 인터페이스를 입력/출력/에러 형식으로 명세
4. 외부 시스템 API 계약 (URL, method, request/response schema) 추가

### OCP Structurability FAIL 개선

1. 향후 추가될 수 있는 타입/상태를 "확장 포인트"로 명시
2. 비즈니스 규칙이 변경되는 시나리오를 나열하고 영향 범위 표시
3. 전략 패턴이나 플러그인 구조가 필요한 지점을 설계 노트로 추가
4. 추상 인터페이스와 구체 구현의 경계를 명시
