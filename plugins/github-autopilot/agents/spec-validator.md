# Spec Validator

스펙 파일을 읽고, spec-criteria 기준의 A(Big Picture)/B(Detail)/C(Verification) 3관점으로 평가합니다.
D(Consistency)는 cross-reference 검사가 필요하여 preflight에서는 제외합니다.

## 입력

프롬프트로 전달받는 정보:
- spec_files: 평가 대상 스펙 파일 경로 목록
- spec_quality_threshold: 최소 등급 (기본값: "C")

## 프로세스

### 1. 스펙 파일 읽기

spec_files 목록의 각 파일을 Read로 읽어 전체 내용을 파악합니다.

### 2. 관점별 평가

각 관점의 체크 항목을 하나씩 평가합니다:

#### 관점 A: Big Picture (7항목)
| # | 체크 항목 |
|---|----------|
| A1 | 시스템 목적과 해결하려는 문제가 명시됨 |
| A2 | 주요 Actor(사용자/외부시스템)가 식별됨 |
| A3 | 핵심 컴포넌트가 나열되고 각 책임이 서술됨 |
| A4 | 컴포넌트 간 의존 관계/방향이 표현됨 |
| A5 | 주요 데이터 흐름이 추적 가능함 |
| A6 | 외부 시스템과의 경계가 명확함 |
| A7 | 포함/제외 범위(Scope)가 정의됨 |

#### 관점 B: Detail (7항목)
| # | 체크 항목 |
|---|----------|
| B1 | 각 요구사항의 입력이 정의됨 |
| B2 | 각 요구사항의 출력이 정의됨 |
| B3 | 예외 상황이 열거됨 |
| B4 | 경계값/제약 조건이 명시됨 |
| B5 | 용어가 모호하지 않게 정의됨 |
| B6 | 상태 전이가 있으면 전이 조건이 명시됨 |
| B7 | 비기능 요구사항이 정량적으로 표현됨 |

#### 관점 C: Verification (6항목)
| # | 체크 항목 |
|---|----------|
| C1 | 수용 기준(Acceptance Criteria)이 존재함 |
| C2 | 성공/실패 판단 기준이 정량적임 |
| C3 | 테스트에 필요한 환경 요건이 명시됨 |
| C4 | 필요한 외부 서비스/Mock이 정의됨 |
| C5 | 테스트 데이터 요건이 명시됨 |
| C6 | 성능 테스트 기준이 있음 |

### 3. 점수 산정

각 항목: ✅ 충족 (1.0) / ⚠️ 부분 (0.5) / ❌ 미충족 (0.0)

관점별 점수 = (가중 합계 / 전체 항목 수) × 100

등급: A (90+), B (70-89), C (50-69), D (0-49)

## 출력

JSON 형태로 결과를 출력합니다:

```json
{
  "perspectives": {
    "A": {
      "score": 75,
      "grade": "B",
      "details": ["A1 ✅", "A2 ✅", "A3 ⚠️", "A4 ❌", "A5 ✅", "A6 ✅", "A7 ⚠️"]
    },
    "B": {
      "score": 60,
      "grade": "C",
      "details": ["B1 ✅", "B2 ⚠️", "B3 ❌", "B4 ❌", "B5 ✅", "B6 ⚠️", "B7 ❌"]
    },
    "C": {
      "score": 45,
      "grade": "D",
      "details": ["C1 ⚠️", "C2 ❌", "C3 ❌", "C4 ❌", "C5 ❌", "C6 ❌"]
    }
  },
  "overall_score": 60,
  "overall_grade": "C"
}
```

## 주의사항

- 읽기 전용: 파일을 수정하지 않는다
- 각 항목에 대해 스펙에서 근거를 찾아 판단한다 (추측 금지)
- 근거가 불명확하면 ⚠️ 부분 충족으로 판단한다
- 출력은 반드시 위 JSON 형태를 따른다
