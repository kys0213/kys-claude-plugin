# interview — brainstorm 모드

막연한 아이디어를 **검증된 설계**로 함께 발전시킨다. 컨텍스트 탐색에서 출발해 접근법을 비교하고, 설계를 섹션 단위로 검증받고, self-review 와 승인 게이트를 거쳐 구현 단계로 넘긴다. `interview` SKILL.md 의 공통 원칙(한 번에 하나, 추천 답 동반, 코드베이스 우선 탐색, 승인 전 구현 금지)을 그대로 따른다.

## 언제 쓰나

- "같이 brainstorm 하자", "설계 같이 잡자", "막연한데 방향부터" 처럼 **아직 구체안이 없는** 작업
- 이미 plan 이 있으면 `grill` 모드로. 의도가 이미 합의됐고 아키텍처 스펙 문서를 잡는 단계면 `spec:design` 으로.

## 프로세스

### 1. 컨텍스트 탐색 (먼저, 직접)
질문하기 전에 코드베이스에서 prior art·관련 모듈·기존 컨벤션·영향 범위를 Grep/Read 로 확인한다. 발견한 사실을 "이미 X 패턴이 있습니다" 형태로 먼저 제시한다.

### 2. 명확화 질문 — 한 번에 하나씩
이해 갭을 메우는 질문을 하나씩 한다. 각 질문에 **추천 답과 근거**를 동반한다. 목적·제약·성공 기준이 드러날 때까지 계속하되, 코드로 답이 정해지는 건 묻지 말고 탐색 결과로 메운다.

### 3. 접근법 2–3개 + trade-off
설계 공간이 좁혀지면 **서로 다른 접근법 2–3개**를 제시한다.
- 각 접근의 핵심 아이디어 / 장점 / 비용·리스크 / 적합 조건
- 그중 **추천안과 이유**를 명시한다. 사용자가 선택하면 다음 단계로.

### 4. 설계 제시 — 섹션 단위 검증
선택된 방향을 **한 번에 쏟지 않고 섹션 단위로** 제시하고, 섹션마다 사용자 검증을 받은 뒤 다음으로 간다. 섹션 구성은 설계 산출물 3요소(인터페이스 정의·모듈 구조·의존성 방향)를 반드시 포함한다:

1. 목적·범위 (비목표 포함)
2. **인터페이스 정의** — 공개 API / 시그니처 / 계약
3. **모듈 구조** — 책임 분리
4. **의존성 방향** — 무엇이 무엇에 의존하는가
5. 에러·엣지 케이스
6. 영향 범위·리스크

### 5. 시각화 (조건부 제안)
텍스트보다 시각 표현이 이해를 명확히 돕는 섹션(구조 비교·흐름·대안 대조)에서는 **먼저 제안하고 동의를 받은 뒤** 사용한다:
- markdown 표 / ASCII·mermaid 다이어그램 — 모듈 구조, 의존성 방향, 흐름
- AskUserQuestion 의 preview — 대안 2–3개를 나란히 비교시킬 때

모든 섹션에 강제하지 않는다. 시각화가 더해주는 게 없으면 텍스트로 간다.

### 6. 설계 self-review
최종 승인을 요청하기 직전 스스로 점검한다:
- 섹션 간 모순은 없는가
- 검증 안 된 가정이 사실처럼 적혀 있지 않은가
- 요청하지 않은 추상화(YAGNI 위반)가 끼어들지 않았는가
- 빠뜨린 엣지·사이드이펙트는 없는가

발견하면 해당 섹션을 수정해 다시 제시한다.

### 7. 승인 게이트 (Hard Gate)
전체 설계 요약을 제시하고 **명시적 승인**을 받는다. 승인 전에는 코드 작성·스캐폴딩·파일 변경을 시작하지 않는다. 작업이 단순해 보여도 동일.

```
## 설계 요약
- 접근: <선택안>
- 인터페이스: <핵심 시그니처/계약>
- 모듈 구조 / 의존성 방향: ...
- 비목표: <안 건드릴 것>
- 리스크: <남은 위험>
→ 이 설계로 갈까요? (승인 / 섹션 수정)
```

### 8. 핸드오프 — 설계 영속화는 기존 채널로
brainstorm 자체는 별도 설계 파일을 만들지 않는다 (스펙 문서는 `spec` 의 책임). 합의된 설계 요약을 들고:
- 코드 변경 작업 → **Plan Mode** — plan 파일이 설계의 캐리어가 된다
- 아키텍처/다중 컴포넌트 규모이거나 스펙 문서로 남겨야 하면 → **`spec:design`** 으로 에스컬레이트

## 종료 조건

- 설계 요약이 승인되어 Plan Mode/`spec` 으로 핸드오프됨
- 또는 사용자가 방향을 보류/중단

## 예시 (탐색 → 접근법 → 섹션별 검증 → 승인)

사용자: "사용자 활동 로그 같이 brainstorm 하자"

```
[탐색] 기존 로깅/이벤트 인프라 확인 → src/events/ 에 EventBus 존재, DB 는 sqlite.
       → "이미 EventBus 가 있어 여기에 얹는 게 자연스럽습니다."

Q1 보존 기간 정책이 있나요?
   추천: 90일 — 유사 테이블(audit_log)이 90일 보존. prior art 와 일치 권장.
   → (사용자: 90일 확정)

접근법 제시:
  A. EventBus 구독자 추가 + 기존 audit_log 테이블 재사용
     +적은 신규 코드 / -스키마가 audit 용이라 필드 안 맞을 수 있음
  B. 전용 activity_log 테이블 신규
     +깔끔한 모델 / -마이그레이션·신규 배선 비용
  추천: A — 요구가 단순 조회 수준이고 EventBus 재사용으로 표면적 최소.
  → (사용자: A 선택)

[섹션별 제시] §인터페이스: ActivityLogger.record(event) 계약 → 검증 →
  §모듈 구조: events/subscribers/activity.rs 단일 책임 → 검증 →
  §의존성 방향: subscriber → EventBus 추상에만 의존 → 검증 → ...
  (의존성 방향은 ASCII 다이어그램 제안 → 동의 시 도식화)

[self-review] "retention 90일 삭제 주체가 미정" 발견 → §에러·엣지에 추가 후 재제시

[설계 요약 제시] → "이 설계로 갈까요?" → (승인) → Plan Mode 핸드오프
```

승인 전에는 어떤 파일도 만들지 않는다.
