# interview — grill 모드

이미 존재하는 plan/design 을 **집요하게 심문**해 공유된 이해에 도달할 때까지 결정 트리의 모든 가지를 해소한다. `interview` SKILL.md 의 공통 원칙(한 번에 하나, 추천 답 동반, 코드베이스 우선 탐색, 승인 전 구현 금지)을 그대로 따른다.

## 언제 쓰나

- 사용자가 "grill me", "이 계획 심문해줘", "내 설계 스트레스 테스트해줘" 라고 요청
- 이미 작성된 plan / 설계 문서 / 접근안이 있고, 빈틈·암묵적 가정·결정 간 충돌을 드러내고 싶을 때

## 프로세스

### 1. 결정 트리 추출
대상 plan/design 을 읽고, 내려야 할 **결정(decision)** 들을 트리로 분해한다.
- 각 노드 = 하나의 결정 (예: "저장소를 Redis 로 할까 in-memory 로 할까")
- 엣지 = 결정 간 의존성 (상위 결정이 하위 선택지를 제약)
- 코드베이스로 답이 정해지는 노드(기존 컨벤션·prior art·제약)는 **질문하지 말고 탐색해 확정**하고 그 사실을 명시한다.

### 2. 가지별 순회 — 한 번에 하나씩
의존성 위상 순서로 루트→리프를 내려가며, 미해소 결정마다:
1. 결정 지점과 선택지를 1–2줄로 제시
2. **추천 답과 근거**를 함께 제시 ("내 추천: X — 이유")
3. 사용자의 답/이견을 받아 확정
4. 그 결정이 하위 가지를 가지치기하면(선택지 제거) 반영하고 다음으로

질문은 **한 번에 하나만**. 답이 다음 가지의 전제가 되므로 묶어서 묻지 않는다.

### 3. 모순·빈틈 적극 노출
- 두 결정이 충돌하면(예: "동기 처리" + "10초 타임아웃 금지") 즉시 표면화하고 어느 쪽을 양보할지 묻는다.
- plan 이 암묵적으로 가정한 것(미언급 전제)을 찾아 명시적 확인을 요청한다.
- "가장 위험한 오해 후보"를 1개 지목하고 그 부분을 먼저 친다.

## 종료 조건

- 트리의 모든 가지가 (탐색 또는 합의로) 해소됨
- 또는 사용자가 "충분하다 / 그만" 선언

## 산출물 — 합의된 결정 목록

```
## 심문 결과 — 합의된 결정
1. <결정 지점>: <확정 선택> — <근거>
2. ...

미해소(보류): <있으면 명시, 없으면 "없음">
다음: 진행 or Plan Mode(코드 변경 설계)로 핸드오프
```

해소가 끝나면 코드 변경이 필요한 경우 Plan Mode 로, 아키텍처 규모면 `spec:design` 으로 핸드오프한다. 합의 전에는 구현을 시작하지 않는다.

## 예시 (한 가지씩, 추천 답 동반)

사용자: "이 캐시 추가 plan 좀 grill 해줘"

```
[탐색] 기존 코드에 cache 추상화가 있는지 먼저 확인 → src/cache/ 없음, HTTP 레이어에 직접 fetch.
       → "현재 캐시 레이어가 없어 신규 도입입니다. 사실로 깔고 갑니다."

Q1 (루트 결정) 캐시 저장소: in-memory vs Redis?
   추천: in-memory — 단일 인스턴스이고 plan 에 분산 요구가 없음. 분산 전이면 Redis.
   → (사용자: in-memory) → 하위 가지 "Redis 연결 설정" 가지치기됨

Q2 무효화 전략: TTL vs 명시적 invalidate?
   추천: TTL — 데이터 신선도 요구가 "분 단위 허용"이라 단순 TTL 로 충분.
   → (사용자: TTL 60s 확정)

[모순 노출] plan 의 "항상 최신 응답 보장" 문구와 TTL 60s 가 충돌.
   → 어느 쪽을 양보할지 확인 요청 → (사용자: 60s stale 허용으로 문구 수정)
```

결과 출력은 위 "합의된 결정 목록" 형식을 따른다.
