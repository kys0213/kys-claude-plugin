# .claude/rules 설계 (rules design)

아키텍처 레이어별 규칙 매핑, `paths:` frontmatter 전략, 규칙 파일 템플릿, 다중 언어 전략, paths 범용화 원칙, 규칙 내용 범용화 원칙. rules 구조를 제안·생성·검증할 때 로드한다.

## 1. 아키텍처 레이어 → 규칙 파일 매핑

### 공통 관심사 (언어 무관)

모든 프로젝트에 적용할 수 있는 보편적 관심사:

| 관심사 | 규칙 파일명 | 핵심 원칙 |
|---|---|---|
| 진입점 (Entrypoint) | `handler.md` 또는 `controller.md` | HTTP/CLI 파싱만, 로직 위임 |
| 비즈니스 로직 | `service.md` | 인터페이스 정의, DI, 트랜잭션 조율 |
| 데이터 접근 | `repository.md` | 추상화, 쿼리 캡슐화 |
| 도메인 모델 | `entity.md` 또는 `model.md` | 불변식, 밸류 오브젝트 |
| DTO / 계약 | `dto.md` | 입출력 타입 분리, 직렬화 |
| 모듈 경계 | `module-boundary.md` | 단방향 의존, 순환 금지 |
| 테스트 | `testing.md` | mock 전략, 테스트 구조 |
| 에러 처리 | `error-handling.md` | 에러 타입, 전파 규칙 |

### 프론트엔드 전용 관심사

| 관심사 | 규칙 파일명 | 핵심 원칙 |
|---|---|---|
| 컴포넌트 | `component.md` | props 설계, 합성, 접근성 |
| 훅/컴포저블 | `hook.md` | 상태 캡슐화, 재사용 |
| 상태 관리 | `store.md` | 전역/로컬 분리, 구독 패턴 |
| 페이지/라우팅 | `page.md` | 데이터 페칭, 레이아웃 |

## 2. `paths:` Frontmatter 전략

### Lazy Context Injection 원칙

규칙 파일은 `paths:` frontmatter를 통해 관련 파일 수정 시에만 Claude 컨텍스트에 로드됩니다. 이를 통해 토큰 낭비를 방지합니다.

```yaml
---
paths:
  - "**/*handler*.go"        # 파일명 패턴
  - "**/*.controller.ts"     # 확장자 + 접미사 패턴
  - "**/controllers/**"      # 디렉토리 패턴
---
```

### 패턴 작성 가이드

위 3가지 패턴(파일명 접미사·파일명 포함·디렉토리 기반) 중 프로젝트의 명명 규칙에 맞는 것을 고르고, 혼용 프로젝트라면 배열로 복합 지정합니다. 명명 규칙이 있는 프레임워크(NestJS 등)는 접미사 패턴이, Go처럼 접미사 규칙이 없는 언어는 파일명 포함 패턴이 적합합니다.

모노레포에서의 범용화 패턴은 §5 (paths 범용화 원칙)를 참조합니다.

## 3. 규칙 파일 템플릿

### 표준 구조

모든 규칙 파일은 다음 구조를 따릅니다:

```markdown
---
paths:
  - "<glob pattern>"
---

# <레이어명> Convention

> 한 줄 요약

## 원칙

1. **원칙 1**: 설명
2. **원칙 2**: 설명

## DO

- 원칙 서술 (+ 필요 시 구조 수준 패턴)

## DON'T

- 원칙 위반 서술 (+ 필요 시 구조 수준 패턴)

## 체크리스트

- [ ] 확인 항목 1
- [ ] 확인 항목 2
```

DO/DON'T 항목에 실제 코드 조각·실제 식별자를 담을지 여부는 §7 (규칙 내용 범용화 원칙)을 따릅니다.

### DO/DON'T 예시 (레이어별)

#### Handler / Controller

```markdown
## DO

- HTTP 파싱과 응답 변환만 담당
- 에러를 적절한 HTTP 상태 코드로 매핑
- 입력 유효성 검사는 DTO/스키마에 위임

## DON'T

- 비즈니스 로직을 핸들러에 직접 구현
- 데이터베이스를 직접 호출
- 다른 핸들러를 호출
```

Service, Repository 등 다른 레이어도 같은 추상화 수준(역할 서술 + 위반 사례)으로 DO/DON'T를 작성합니다 — 예: Service는 "인터페이스 우선 정의·생성자 주입" DO와 "HTTP 객체 의존·구체 구현 의존" DON'T, Repository는 "데이터 접근만 캡슐화" DO와 "비즈니스 로직 포함" DON'T.

## 4. Gap 분석 모드

기존 `.claude/rules/` 파일이 있을 때 gap 분석을 수행합니다.

### 분석 기준

1. **누락된 레이어**: 코드베이스에 해당 레이어가 존재하지만 규칙이 없는 경우
2. **paths 미설정**: 규칙 파일에 `paths:` frontmatter가 없어 항상 로드되는 경우
3. **과도한 범위**: `paths:` 패턴이 너무 넓어 불필요하게 자주 로드되는 경우
4. **언어 미지원**: 모노레포에서 특정 언어의 규칙이 누락된 경우
5. **구현 결합**: 규칙에 실제 코드 조각·실제 식별자가 포함되어 코드 변경 시 규칙이 stale해질 위험이 있는 경우

---

## 5. 다중 언어/프레임워크 레포 전략

### Prefix 전략

2개 이상의 언어/프레임워크가 혼용된 레포에서는 prefix로 구분합니다.

### 공통 규칙 vs 언어별 규칙

- **공통**: `module-boundary.md`, `error-handling.md`, `testing.md`
- **언어별**: prefix를 붙여 분리 (`go-`, `ts-`, `py-`, `rs-`)

## 6. paths 범용화 원칙

### 핵심 원칙

paths는 **레이어 구조**(역할)를 표현해야 하며, **위치**(컨테이너 경로)를 표현해서는 안 됩니다.

### 판별 기준

- **레이어**: 파일의 역할을 나타내는 디렉토리 (`core/`, `commands/`, `handlers/`, `service/` 등)
- **위치**: 레이어가 속한 컨테이너 (`plugins/atelier/`, `packages/api/`, `apps/web/` 등)
- paths에는 레이어만 남기고, 위치는 `**/`로 대체합니다.

### 범용화 알고리즘

1. **레이어 식별**: 수집된 경로에서 역할을 나타내는 디렉토리를 구분
2. **컨테이너 추상화**: 레이어 상위의 컨테이너 경로를 `**/`로 대체
3. **교차 검증**: 범용 패턴이 동일 레이어의 파일만 매칭하는지 Glob 테스트
4. **범위 조정**: 의도하지 않은 파일이 포함되면 패턴을 좁히거나 예외 명시

### 예시 (원칙 적용)

| 수집된 경로 | 레이어 | 컨테이너 | 범용 패턴 |
|---|---|---|---|
| `plugins/atelier/src/core/git.ts` | `core/` | `plugins/atelier/src/` | `**/core/**/*.ts` |

### 예외 (범용화하지 않는 경우)

의도적으로 범위를 제한해야 하는 paths는 특정 경로를 그대로 사용합니다:

- 자동 생성 코드 (`generated/`, `__generated__/`)
- 마이그레이션 파일 (`migrations/`)
- 루트 설정 파일 (`*.config.ts`, `*.toml`)

### 체크리스트

- [ ] paths가 레이어(역할)를 표현하는가, 위치(컨테이너)를 표현하는가?
- [ ] 컨테이너 경로가 `**/`로 대체되었는가? (예외 제외)
- [ ] 동일 레이어가 복수 위치에 존재할 때 하나의 패턴으로 통합되었는가?
- [ ] Glob 테스트로 의도한 파일만 매칭되는지 확인했는가?
- [ ] 의도적 범위 제한이 필요한 경우 예외로 명시했는가?

## 7. 규칙 내용 범용화 원칙

### 핵심 원칙

규칙 본문은 **역할·계약·불변식**을 표현해야 하며, **현재 구현 스냅샷**(실제 코드 조각·실제 식별자·실제 경로)을 표현해서는 안 됩니다.

### 판별 기준

"구현 디테일이 바뀌면 이 문장이 깨지는가?"를 묻습니다. 깨진다면 일반화 대상입니다.

### 예시 수준 규정

- **허용**: 의사코드, 형태 패턴 (예: `findBy<조건>` 형태 표기)
- **금지**: 실제 파일에서 복사한 코드 블록, 실제 클래스명, 실제 경로 인용

### 예외 (구체 표기를 허용하는 경우)

**형식이 곧 계약**인 경우는 구체 표기를 허용합니다. frontmatter 스키마, CLI 출력 포맷 등은 구현 디테일이 아니라 스펙 그 자체이기 때문입니다.

### 체크리스트

- [ ] 규칙 문장이 구현이 아닌 역할·계약·불변식을 표현하는가?
- [ ] "구현이 바뀌면 깨지는 문장"이 남아있지 않은가?
- [ ] 예시가 의사코드·형태 패턴 수준을 넘어 실제 코드를 복붙하지 않았는가?
- [ ] 실제 클래스명·실제 경로가 원칙 설명이 아닌 예시로 등장하지 않는가?
- [ ] 형식 자체가 계약인 예외 항목만 구체 표기를 사용했는가?
