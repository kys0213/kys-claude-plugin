# report

작업 결과·조사 내용·진행 상황을 **사람이 읽고 의사결정할 수 있는 마크다운 보고서**로 형식화하는 범용 스킬입니다. 무엇을 보고할지(내용)는 이미 정해져 있고, 여기서는 그것을 *청중에게 맞는 구조와 깊이로 정리*합니다. 메인 에이전트가 직접 작성합니다.

## 진입 라우팅 (의도 → 보고서 유형)

| 사용자 의도 (예) | 유형 | 핵심 질문 |
|---|---|---|
| "작업 결과 정리", "PR 요약 보고", "뭐 했는지 보고서로" | 작업 완료 보고 | 무엇을·왜 했고, 어떻게 검증했나 |
| "분석 결과 보고서", "조사한 거 정리", "현황 분석" | 분석/조사 보고 | 무엇을 발견했고, 근거는 무엇인가 |
| "진행 상황 요약", "주간 리포트", "어디까지 됐나" | 진행 상황 보고 | 완료·진행·남은 것, 리스크는 무엇인가 |

> 유형이 애매하면 사용자에게 묻지 말고 **요청 맥락에서 가장 가까운 유형**을 골라 작성하되, 어느 유형으로 썼는지 한 줄로 밝힙니다. 보고 대상 데이터(작업 내역·분석 결과 등)는 입력으로 함께 오면 그대로 쓰고, 비어 있으면 작성하지 않고 무엇이 필요한지 되묻습니다 — 없는 내용을 추측해 채우지 않습니다.

## 작성 원칙

1. **BLUF (Bottom Line Up Front)** — 결론을 맨 앞에 둔다. 읽는 사람이 첫 3줄만 읽어도 핵심을 알 수 있어야 한다. 서론·과정부터 늘어놓지 않는다.
2. **근거 명시 (Evidence-backed)** — 모든 주장에 근거를 단다. 파일 경로·라인, 명령 출력, 수치, 링크 등 검증 가능한 출처를 붙인다. "잘 동작한다"가 아니라 "테스트 N건 통과(`cargo test` 출력)"로 쓴다.
3. **사실과 판단 분리** — 관찰한 사실과 그에 대한 해석·권고를 구분해 쓴다. 추정은 "추정", 미확인은 "미확인"으로 명시한다.
4. **청중 적합성 (Audience-fit)** — 읽는 사람이 누구인지에 따라 깊이를 조절한다. 의사결정자에게는 결론·영향·선택지를, 동료 개발자에게는 변경 지점·근거를 우선한다.
5. **간결성 (Concise)** — 한 문장 한 정보. 같은 내용을 표·목록으로 표현할 수 있으면 산문 대신 표를 쓴다. 불필요한 수식어를 빼고 명사형·능동형으로 쓴다.

## 작성 프로세스

```
1. 청중·목적 확정 → 누가 읽고, 무슨 결정을 내리려 하는가
2. 보고 유형 선택   → 작업 완료 / 분석 / 진행 상황
3. 골격 작성        → 요약(결론)을 먼저 한 문단으로 쓴다
4. 본문 채우기      → 근거와 함께 사실 → 판단 순으로 정리
5. 검증            → 모든 주장에 근거가 붙었는지, 결론과 본문이 일치하는지 확인
```

- 3단계의 **요약을 가장 먼저 쓴다**. 요약을 먼저 쓰면 본문이 결론을 향해 정렬된다. 본문을 다 쓴 뒤 요약과 어긋나면 요약이 아니라 본문을 의심한다.
- 보고 대상이 크면 본문을 섹션으로 쪼개되, 요약은 항상 한 화면 안에 들어오게 유지한다.

## 범용 보고서 구조

모든 유형이 공유하는 골격입니다. 해당 없는 섹션은 생략합니다 (빈 섹션을 남기지 않는다).

| 섹션 | 목적 | 필수 |
|---|---|---|
| 제목 | 무엇에 대한 보고인지 한 줄 | 필수 |
| 요약 (TL;DR) | 결론·핵심 결과를 3~5줄로 | 필수 |
| 배경/목적 | 왜 이 작업·조사를 했는지 | 권장 |
| 내용 | 한 일 / 발견 / 진행 현황 (유형별 본문) | 필수 |
| 결과·근거 | 검증 방법과 그 결과, 출처 | 필수 |
| 리스크/미해결 | 남은 문제, 한계, 가정 | 권장 |
| 다음 단계 | 후속 조치·권고·의사결정 요청 | 권장 |

### 유형별 "내용" 섹션 변형

- **작업 완료 보고**: 변경 사항(무엇을), 이유(왜), 영향 범위(어디에), 검증(어떻게 확인)
- **분석/조사 보고**: 발견 사항(사실), 해석(의미), 근거(출처) — 중요도 순 정렬
- **진행 상황 보고**: ✅ 완료 / 🔄 진행 중 / ⏳ 예정 / 🚧 블로커 로 구분

## 출력 형식

````markdown
# [보고서 제목]

> 유형: 작업 완료 보고 · 작성: 2026-06-17 · 대상: <청중>

## 요약
- **결론**: <한 문장 핵심>
- 핵심 결과 2~3개를 불릿으로
- 의사결정 요청이 있으면 여기에 한 줄

## 배경
<왜 이 작업을 했는지 1~2문단. 불필요하면 생략>

## 내용
<유형별 본문. 표나 목록을 우선 사용>

| 항목 | 변경/발견 | 근거 |
|------|-----------|------|
| ... | ... | `path/to/file.rs:42` |

## 결과·근거
- 검증 방법: <명령/절차>
- 결과: <출력 요약, 수치>

## 리스크 / 미해결
- <남은 문제 또는 "없음">

## 다음 단계
- [ ] <후속 조치 또는 권고>
````

진행 상황 보고 예시(내용 섹션):

```markdown
## 진행 현황
- ✅ 완료: 인증 모듈 구현, 테스트 12건 통과
- 🔄 진행 중: API 연동 (70%)
- ⏳ 예정: 문서화
- 🚧 블로커: 외부 API 키 발급 대기 (담당: 인프라팀)
```

## 품질 체크리스트

보고서를 내보내기 전에 확인합니다:

- [ ] 첫 3~5줄(요약)만 읽어도 결론을 알 수 있는가?
- [ ] 모든 주장에 검증 가능한 근거(경로·수치·출력·링크)가 붙었는가?
- [ ] 사실과 판단·추정이 구분되어 있는가?
- [ ] 빈 섹션이나 의미 없는 수식어가 없는가?
- [ ] 결론과 본문이 서로 모순되지 않는가?
- [ ] 청중이 다음에 무엇을 해야 하는지(다음 단계) 명확한가?
