# report-write

보고서를 **사람이 신뢰하고 빠르게 이해할 수 있게** 쓰기 위한 작문 기준입니다. 보고서의 구조는 상황(청중·목적·분량)에 따라 매번 다르므로 고정 양식을 강제하지 않습니다. 대신 어떤 보고서에도 적용되는 **문체·용어·청중 적응 기준**을 제공합니다. 메인 에이전트가 직접 작성합니다.

## 작문 철학

1. **신뢰 우선** — 보고서의 가치는 읽는 사람이 그 내용을 *믿고 의사결정에 쓸 수 있는가*에 달렸다. 모든 작문 선택(문체·용어·구조)은 신뢰성과 이해 속도를 높이는 방향으로 한다.
2. **결론 먼저 (BLUF)** — 핵심 결론을 맨 앞에 둔다. 읽는 사람이 첫 문단만 읽어도 결정을 내릴 수 있어야 한다.
3. **사실과 판단 분리** — 관찰한 사실, 그에 대한 해석, 권고를 구분해 쓴다. 추정은 "추정", 미확인은 "미확인"으로 표시한다.
4. **근거는 검증 가능하게** — 주장에는 출처(경로·라인·명령 출력·수치·링크)를 붙인다. "잘 된다"가 아니라 "테스트 12건 통과(`cargo test`)"로 쓴다.

## 문체 (Style)

### 톤
- **객관적·중립적**으로 쓴다. 감정·과장("놀랍게도", "완벽하게")을 빼고 사실로 말한다.
- 자신 없는 내용은 단정하지 말고 불확실성을 그대로 드러낸다 ("~로 보인다", "미확인").

### 문장
- **한 문장 한 정보.** 접속사로 이어붙인 긴 문장은 끊는다.
- **능동형·명사형 우선.** "수정이 이루어졌다" → "X를 수정했다".
- **수동태·이중부정·완곡어법 지양.** "되지 않는 것은 아니다" → "된다" 또는 "안 된다".
- 표·목록으로 표현되는 정보는 산문으로 늘어놓지 않는다.

### 어미·시제 (개조식 vs 서술식)
- 한 보고서 안에서 **어미를 통일**한다. 짧은 항목 나열은 개조식("~함", "~완료"), 설명이 필요한 본문은 서술식("~했다")을 쓰되 둘을 한 섹션에서 섞지 않는다.
- 완료된 일은 과거형, 현재 상태는 현재형, 계획은 미래형으로 시제를 사실에 맞춘다.

```
❌ 인증 모듈을 구현하였고, 이로 인해 보안성이 크게 향상되는 결과가 도출되었음.
✅ 인증 모듈을 구현했다. 로그인 실패율이 측정 기준 0건으로 떨어졌다(테스트 12건).
```

## 용어 해설 기준 (Terminology)

용어를 풀어쓸지 말지는 **청중이 그 용어를 이미 아는가**로 결정한다.

| 상황 | 처리 |
|---|---|
| 청중이 모를 수 있는 전문용어 | **첫 등장 시 한 번** 짧게 해설하고 이후 그대로 사용 |
| 약어·이니셜리즘 | 처음에 풀어쓰고 괄호로 약어 — "지속적 통합(CI)", 이후 "CI" |
| 프로젝트 내부 코드명·은어 | 외부 청중 대상이면 일반 용어로 바꾸거나 정의를 단다 |
| 한 개념을 가리키는 여러 단어 | **하나로 통일**한다. 동의어 혼용은 같은 것을 다른 것으로 오해하게 만든다 |

- **정의 위치**: 용어가 1~2개면 인라인(괄호·각주)으로, 여러 개면 문서 앞이나 끝에 **용어집** 섹션으로 모은다.
- **과잉 해설 금지**: 청중이 당연히 아는 용어(개발자에게 "함수", "커밋")는 해설하지 않는다 — 신뢰와 가독성을 모두 떨어뜨린다.

```
❌ N+1 문제가 있었다. (DB 비전문가 청중에게 무해설)
❌ 함수(특정 작업을 수행하는 코드 묶음)를 추가했다. (개발자 청중에게 과잉)
✅ 반복 조회로 쿼리가 행 수만큼 발생하는 N+1 문제가 있었다. (필요한 곳만, 한 번)
```

## 청중별 적응 (구조는 청중이 결정한다)

고정 양식 대신, **읽는 사람이 무엇을 결정하려 하는지**에 맞춰 무엇을 앞에 둘지 정한다.

| 청중 | 우선 정보 | 깊이 |
|---|---|---|
| 의사결정자 | 결론 · 영향 · 선택지 · 비용 | 근거는 요약, 상세는 부록/링크 |
| 동료 개발자 | 변경 지점 · 근거 · 검증 방법 | 경로·라인 수준까지 |
| 미래의 나/팀 (기록용) | 맥락 · 결정 이유 · 당시 제약 | 재현 가능할 만큼 |

- 같은 사실도 청중에 따라 **분량과 순서**를 바꾼다. 의사결정자에게 코드 diff를 늘어놓거나, 개발자에게 "잘 됨"만 보고하지 않는다.
- 청중이 불명확하면 가장 좁은 청중(가장 많은 맥락이 필요한 쪽)을 기준으로 쓰고, 보고서 머리에 대상 청중을 한 줄 명시한다.

## 구조는 가이드라인 (강제 아님)

막막할 때 쓰는 **출발점**이지 의무 양식이 아니다. 상황에 맞지 않는 섹션은 버리고, 필요한 섹션은 더한다.

- **요약(결론)** — 거의 항상 맨 앞에 둔다. 이것만은 생략하지 않는다.
- 그 외 배경 · 내용 · 결과/근거 · 리스크 · 다음 단계는 **필요할 때만**. 짧은 보고는 요약 + 근거 두 단락으로 충분할 수 있다.
- 진행 상황은 상태 마커로 구분하면 빠르게 읽힌다: ✅ 완료 · 🔄 진행 중 · ⏳ 예정 · 🚧 블로커.
