# Big Picture 설계 커맨드 (/design)

AI가 먼저 제안하고, 사용자가 피드백/의사코드로 방향을 잡으며, AI가 반영 후 다른 관점에서 도전하는 **핑퐁 설계 세션**입니다.

> 핑퐁 루프·6개 관점·수렴 판단·안티패턴 등 대화 프로토콜은 `spec-workflow` skill 의 `references/design-protocol.md` 에 있습니다. 이 커맨드는 진입점과 **Big Picture 고유의 출력 구조**만 담습니다.

## 사용법

```bash
/design
/design "사용자 인증 시스템"
/design "결제 파이프라인 리팩토링"
```

| 인자 | 필수 | 설명 |
|------|------|------|
| 주제 | No | 설계 대상 주제 (미지정 시 대화로 도출) |

## 세션 시작

주제가 없으면 **AskUserQuestion** 으로 주제를 먼저 묻는다. 주제가 주어지면:

1. 기존 코드베이스가 있으면 Glob/Read 로 현재 구조를 파악한다
2. 시스템의 목적과 경계를 **구체적으로 제안**한다 (질문이 아닌 의견이 담긴 1-2문단)
3. **AskUserQuestion** 으로 제안에 대한 피드백을 요청한다

이후 `spec-workflow` `references/design-protocol.md` 의 핑퐁 루프를 수렴까지 반복한다.

## 깊이 기준 (Big Picture)

- **포함**: 목표, 설계 철학(의견), 컴포넌트 목록과 책임, 컴포넌트 간 데이터 흐름, 확장 가능 지점
- **제외**: trait/인터페이스 시그니처, 의사코드, 상태 전이 다이어그램, DB 스키마, API 상세 → `/design-detail` 의 영역

## 출력 구조

사용자가 정리를 승인하면, 합의된 내용을 아래 구조의 마크다운으로 작성한다. 저장 경로는 AskUserQuestion 으로 확인 (기본: `spec/DESIGN.md`) 후 Write.

```markdown
---
related_paths:
  - {추정 코드 경로 1}
  - {추정 코드 경로 2}
---

# DESIGN

> **Date**: {오늘 날짜}
> **Status**: Draft

## 목표

{1-3문장: 이 시스템이 해결하는 문제, 설계 목표로 서술}

## 설계 철학

{번호 매긴 원칙들. 각 1-2문장. 서술이 아닌 의견/결정.}

### 1. {원칙 이름}
{설명}

## 전체 구조

{ASCII 다이어그램: 컴포넌트 배치와 데이터 흐름}

## 관심사 분리

| 레이어 | 책임 | 비고 |
|--------|------|------|
| ... | ... | ... |

## OCP 확장점

{새로운 타입/시스템 추가 시 코어 변경 없이 확장 가능한 지점}

```
새 X = Y impl 추가 → 코어 변경 0
```

## 미결정 사항

{대화 중 확정되지 않은 항목. /design-detail 에서 구체화할 후보}

## 상세 문서

| 문서 | 설명 |
|------|------|
| [concerns/...](...) | ... |
| [flows/...](...) | ... |

> `/design-detail` 로 컴포넌트별/시나리오별 상세 설계를 진행할 수 있습니다.
```

> frontmatter `related_paths` 채우는 기준은 `references/design-protocol.md` 참조. 신규 설계라 코드가 아직 없으면 비워둔다.

## 주의사항

- 사용자가 이전 논의를 번복해도 자연스럽게 수용한다
- 기존 코드를 참조하되 설계가 기존 구조에 종속되지 않도록 한다
- 최종 승인 전까지 파일 저장 안 함
