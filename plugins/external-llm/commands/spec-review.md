# 스펙 리뷰 커맨드 (/spec-review)

4가지 관점 × 3개 LLM으로 스펙/코드를 종합적으로 리뷰합니다.
각 관점마다 독립적인 Claude 인스턴스(Agent Teams teammate)가 뜨고,
내부에서 Claude + Codex + Gemini 3개 LLM을 호출하여 컨센서스를 도출합니다.

## 4가지 리뷰 관점

| 관점 | 설명 |
|------|------|
| 사이드이펙트 분석 | 변경으로 인해 영향받는 기존 코드, 의존성, 동작을 사전에 파악 |
| 단순성 & 유지보수성 | 불필요한 복잡도 없이 단순하면서 유지보수하기 좋은 구조인지 |
| 컨벤션 부합성 | 기존 프로젝트 컨벤션(CLAUDE.md, 코딩 스타일, 아키텍처 패턴)에 맞는지 |
| 테스트 계획 | 테스트 계획이 존재하는지, 없으면 QA 관점의 테스트 계획 제안 |

## 사용법

```bash
# 파일 패턴 지정
/spec-review "src/**/*.rs"
/spec-review "plugins/external-llm/**/*.md"

# 추가 컨텍스트
/spec-review "src/**/*.rs" "인증 모듈 리팩토링 관련"

# 자연어
/spec-review "새로 추가한 Rust 모듈 스펙 리뷰해줘"
```

## 핵심 워크플로우

```
1. Glob으로 파일 경로 수집 (내용 안 읽음!)
2. Agent Teams 활성 여부 확인
3-A. Agent Teams → 4 teammates 생성 (각 관점 1명)
3-B. Agent Teams 미활성 → Task 4개 병렬 (fallback)
4. 각 teammate/subagent가 내부에서 3-LLM 호출
5. 결과 취합 → 종합 리포트
```

## 작업 프로세스

### Step 1: 사용자 요청 파싱

사용자 요청에서 추출:
- **대상 파일**: glob 패턴 또는 키워드
- **컨텍스트**: 추가 요청사항 (어떤 기능/변경에 대한 리뷰인지)

**파일 패턴 매핑**:
- "src" / "코드" → `src/**/*.{ts,tsx,js,rs,py}`
- "plans" / "설계" → `plans/*.md`
- "docs" / "문서" → `docs/**/*.md`
- "plugins" → `plugins/**/*.md`
- Glob 패턴이 직접 제공되면 그대로 사용

### Step 2: Glob으로 파일 경로만 수집

**CRITICAL**: 파일 내용을 읽지 않습니다! 경로만 수집합니다.

```
Glob: src/**/*.rs
→ ["src/main.rs", "src/lib.rs", ...]
```

파일이 없으면 즉시 에러:
```
Error: '[패턴]'에 맞는 파일을 찾을 수 없습니다.
```

### Step 3: 프로젝트 컨벤션 파일 경로 수집

컨벤션 분석을 위해 프로젝트 규칙 파일 경로도 수집합니다:

```
Glob: .claude/rules/*.md → 규칙 파일 경로 목록
CLAUDE.md → 프로젝트 원칙 경로
```

### Step 4: Agent Teams 또는 Subagent 분기

#### 4-A: Agent Teams 사용 (권장)

Agent Teams가 활성화된 경우 에이전트 팀을 생성합니다:

```
에이전트 팀을 만들어 다음 4가지 관점으로 스펙 리뷰를 수행해주세요.

- [파일 경로 목록]

컨벤션 파일:
- CLAUDE.md
- [.claude/rules/*.md 파일 목록]

#### 4-B: Subagent Fallback

Agent Teams가 비활성이면 Task 4개를 병렬 실행합니다:

```
Task(prompt="[관점별 프롬프트]", run_in_background=true)  # x4
```

### Step 5: 결과 취합 및 종합 리포트

4개 관점의 결과를 취합하여 종합 리포트를 작성합니다.

## 관점별 Teammate 프롬프트

### Teammate 1: 사이드이펙트 분석가

```
당신은 사이드이펙트 분석 전문가입니다. 코드/스펙 변경이 기존 시스템에 미치는 영향을 분석합니다.

## 담당 관점: 사이드이펙트 분석

## 분석 기준

1. **직접 영향 (Direct Impact)**
   - 변경된 함수/모듈을 호출하는 코드
   - 변경된 타입/인터페이스에 의존하는 코드
   - 변경된 설정에 영향받는 동작

2. **간접 영향 (Indirect Impact)**
   - 의존성 체인을 통한 전파 (A → B → C)
   - 런타임 동작 변경 (성능, 메모리, 에러 핸들링)
   - 데이터 흐름 변경

3. **숨겨진 영향 (Hidden Impact)**
   - 암묵적 계약 위반 (LSP 위반)
   - 글로벌 상태 변경
   - 스레드 안전성, 동시성 이슈
   - 환경 의존적 동작 변경

4. **위험도 평가**
   - Critical: 기존 기능이 깨질 수 있음
   - Warning: 동작이 미묘하게 변할 수 있음
   - Info: 알아두면 좋은 영향

## 분석 절차

1. 대상 파일을 Read로 읽기
2. 직접 Claude 분석 수행
3. Codex에 동일한 분석 요청:
   bash ${CLAUDE_PLUGIN_ROOT}/../../common/scripts/call-codex.sh "[프롬프트]"
   결과 파일 Read
4. Gemini에 동일한 분석 요청:
   bash ${CLAUDE_PLUGIN_ROOT}/../../common/scripts/call-gemini.sh "[프롬프트]"
   결과 파일 Read
5. 3-LLM 컨센서스 도출

### Teammate 2: 단순성 & 유지보수성 분석가

```
당신은 코드 단순성과 유지보수성 전문가입니다. 불필요한 복잡도를 찾아내고 유지보수하기 좋은 구조를 제안합니다.

## 담당 관점: 단순성 & 유지보수성

## 분석 기준

1. **복잡도 분석**
   - 불필요한 추상화 계층
   - 과도한 제네릭/매크로 사용
   - 깊은 중첩 (if/match/loop)
   - 함수/메서드 길이와 책임 범위

2. **가독성**
   - 네이밍의 명확성
   - 코드 흐름의 직관성
   - 매직 넘버/매직 스트링
   - 주석의 적절성 (과잉 vs 부족)

3. **유지보수 용이성**
   - 변경 시 수정해야 할 지점 수
   - 모듈 간 결합도 (coupling)
   - 테스트 가능성 (testability)
   - 에러 처리의 일관성

4. **SOLID 원칙 준수**
   - SRP: 하나의 모듈이 여러 책임을 갖고 있지 않은지
   - OCP: 새로운 타입 추가 시 기존 코드 수정이 필요한지
   - DIP: 구체 구현에 직접 의존하고 있지 않은지

## 분석 절차

1. 대상 파일을 Read로 읽기

### Teammate 3: 컨벤션 부합성 분석가

## 담당 관점: 컨벤션 부합성

## 분석 기준

1. **프로젝트 규칙 준수**
   - CLAUDE.md에 정의된 설계 원칙 (Design First, SOLID, TDD)
   - .claude/rules/ 하위 규칙들

   - 네이밍 컨벤션 (snake_case, CamelCase 등)
   - 에러 처리 패턴 (Result, Option, panic 사용 방식)

   - 파일/모듈 구조

3. **아키텍처 패턴**
   - 레이어드 아키텍처 준수 (Controller → Service → Domain)
   - 에이전트 설계 원칙 (agent-design-principles.md)
   - 토큰 관리 원칙

4. **일관성**
   - 기존 유사 코드와의 패턴 일치
   - 기존 테스트 작성 스타일과의 일치

### Teammate 4: 테스트 계획 분석가

```
당신은 QA/테스트 전문가입니다. 테스트 계획의 존재 여부를 확인하고, 없다면 스펙에 맞는 테스트 계획을 제안합니다.

## 담당 관점: 테스트 계획

## 분석 기준

1. **기존 테스트 존재 여부**
   - 단위 테스트 (unit test) 존재 확인
   - 통합 테스트 (integration test) 존재 확인
   - E2E 테스트 존재 확인
   - 테스트 커버리지 수준 추정

2. **테스트 품질 평가** (기존 테스트가 있는 경우)
   - 경계값 테스트 (edge cases)
   - 에러 경로 테스트 (error paths)
   - 정상 경로 테스트 (happy paths)
   - 테스트 격리성 (independence)
   - Mock/Stub 적절성

3. **테스트 계획 제안** (기존 테스트가 부족한 경우)
   - TDD 원칙에 따른 테스트 우선순위
   - 블랙박스 테스트 시나리오
   - 인메모리 구현체 활용 전략
   - 핵심 시나리오 목록

4. **QA 관점 체크리스트**
   - 회귀 테스트 필요 여부
   - 성능 테스트 필요 여부
   - 보안 테스트 필요 여부
   - 호환성 테스트 필요 여부

## 컨센서스 분석 기준

- **3/3 합의**: 높은 신뢰도 → 반드시 반영

## Fallback: Subagent 모드

Agent Teams가 비활성인 경우, 4개의 Task를 `run_in_background=true`로 병렬 실행합니다.
각 Task에 관점별 프롬프트를 전달하며, 프롬프트에 다음을 포함합니다:

1. 대상 파일 경로 (Read로 읽을 것)
2. 관점별 분석 기준
3. call-codex.sh, call-gemini.sh 호출 지침
4. 3-LLM 컨센서스 출력 형식

## 주의사항

- **Agent Teams 필요**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 설정 권장
- **API 필요**: Codex, Gemini CLI 설치 필요
- **토큰 최적화**: MainAgent는 파일 경로만 수집, Teammate/Subagent가 실제 내용 읽기
- **Codex/Gemini 불가 시**: 해당 LLM 결과는 "N/A"로 표시, 가용한 LLM으로만 리포트 생성
