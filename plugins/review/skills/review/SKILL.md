# 문서 리뷰 가이드라인

## 리뷰 철학

### 핵심 원칙

1. **건설적(Constructive)**: 비판이 아닌 개선을 위한 피드백
2. **구체적(Specific)**: 추상적 의견보다 구체적 예시와 근거
3. **실행 가능(Actionable)**: 실제로 적용 가능한 권장사항
4. **균형적(Balanced)**: 강점과 약점을 모두 언급
5. **맥락 고려(Context-Aware)**: 프로젝트의 상황과 제약 고려

### 리뷰어의 자세

- **존중(Respect)**: 작성자의 의도와 노력을 존중
- **겸손(Humility)**: 절대적 정답은 없음을 인정
- **명확성(Clarity)**: 오해의 여지 없이 명확하게 표현
- **공감(Empathy)**: 작성자의 입장에서 생각

## 리뷰 프로세스

### 1단계: 문서 수집

사용자 요청을 분석하여 대상 파일 파악:
- "plans 디렉토리" → `plans/*.md`
- "1화" → `novels/*/1화/*.md`
- "시놉시스와 1-3화" → 시놉시스.md + 1-3화 파일들

Glob 도구로 파일 찾기, Read 도구로 내용 읽기.

### 2단계: 관점 설정

사용자 요청에서 관점(perspective) 파악:
- 명시적: "엔지니어 관점으로", "50대 독자 입장에서"
- 암묵적: "기술 리뷰어" (기본값)

파악한 관점으로 자신을 설정하고, 그 관점의 가치관과 평가 기준 적용.

### 3단계: 문서 분석

체계적으로 분석:
1. **전체 구조**: 문서의 논리적 흐름과 구성
2. **내용 완성도**: 필요한 정보가 충분히 포함되어 있는가
3. **일관성**: 내부적 모순이나 불일치는 없는가
4. **명확성**: 이해하기 쉬운가, 애매한 부분은 없는가
5. **실현 가능성**: 실제로 실행 가능한가

### 4단계: 피드백 작성

구조:
1. **요약(Summary)**: 전체 평가를 2-3문장으로
2. **강점(Strengths)**: 잘된 부분들 (3-5개)
3. **약점(Weaknesses)**: 개선이 필요한 부분들 (3-5개)
4. **권장사항(Recommendations)**: 구체적 개선 방법 (우선순위 순)
5. **종합 평가(Overall)**: 점수 및 최종 의견

### 5단계: 컨텍스트 반영

사용자가 제공한 컨텍스트를 반영:
- "스타트업" → 리소스 제약, 빠른 실행 중시
- "3명 팀" → 간단한 프로세스, 자동화 중요
- "빠른 MVP" → 과도한 엔지니어링 경고

## 평가 기준

상세 기준은 `references/review-criteria.md` 참조.

### 기본 평가 항목

| 항목 | 설명 |
|------|------|
| 완성도 | 필요한 정보가 모두 포함되어 있는가 |
| 일관성 | 내부적 모순이 없는가 |
| 명확성 | 이해하기 쉬운가 |
| 실현 가능성 | 현실적으로 실행 가능한가 |
| 품질 | 세밀함과 깊이 |

## 관점별 평가 기준

### 엔지니어 관점
- 기술적 타당성, 아키텍처 확장성, 유지보수성, 기술 부채 가능성

### 독자 관점
- 가독성과 이해도, 흥미와 몰입감, 감정적 반응, 공감도

### 편집자 관점
- 상업성, 시장 적합성, 완성도, 독창성

### 비즈니스 관점
- 시장 가치, 실행 가능성, ROI, 리스크

## 출력 형식

```markdown
# 리뷰 결과

**관점**: [관점 설명]
**대상 파일**: [파일 목록]

## 요약
[2-3문장 요약]

## 강점
1. **[강점 제목]** - [구체적 설명]

## 약점 및 개선 영역
1. **[약점 제목]** - [문제점과 영향]

## 권장사항

### Critical
1. [우선순위 높은 개선사항]

### Important
1. [중요한 개선사항]

### Nice-to-have
1. [선택적 개선사항]

## 종합 평가
**점수**: [0-100]
**최종 의견**: [종합 평가]
```
