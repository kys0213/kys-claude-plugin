# 좋은 리뷰 예시

이 문서는 고품질 리뷰의 예시를 보여줍니다.

---

## 예시 1: 기술 계획 리뷰 (엔지니어 관점)

### 요청
> "staff+ 엔지니어 관점으로 plans 디렉토리를 리뷰해줘. 우리는 3명 스타트업 팀이고 빠른 MVP가 필요해"

### 리뷰 결과

**관점**: Staff+ 소프트웨어 엔지니어 (10년+ 경력)
**대상 파일**: plans/*.md
**컨텍스트**: 3명 스타트업 팀, 빠른 MVP 필요

---

## 📋 요약

전반적으로 매우 실용적이고 체계적인 계획입니다. Phase 구분이 명확하고 우선순위 설정이 합리적입니다. 다만 스타트업 환경을 고려할 때 일부 과도한 엔지니어링이 우려되며, MVP에 집중하기 위한 범위 축소를 권장합니다.

---

## ✅ 강점

### 1. **명확한 Phase 구분과 우선순위**
- Phase 0-4로 명확히 분리되어 있어 순차적 실행이 용이
- Phase 0(리뷰 시스템)을 최우선으로 두어 품질 보증 인프라를 먼저 구축하는 전략이 탁월
- 각 Phase의 목표와 산출물이 구체적으로 정의됨
- **영향**: 팀이 무엇을 언제 해야 할지 명확히 알 수 있음

### 2. **자연어 인터페이스 채택**
- 플래그 기반 (`--as`, `--files`) 대신 자연어를 사용
- 사용자 경험이 크게 개선됨: `/review-claude "staff+ 엔지니어 관점으로 plans를 리뷰해줘"`
- LLM의 지능을 완전히 활용하여 관점, 파일, 컨텍스트를 자동 파악
- **영향**: 학습 곡선이 낮고 사용이 직관적

### 3. **다중 LLM 활용 전략**
- Claude, OpenAI Codex, Gemini를 모두 활용하여 다양한 관점 확보
- 백그라운드 실행으로 토큰 효율성 극대화
- JSON 기반 통신으로 결과 파싱 용이
- **영향**: 단일 LLM 편향을 피하고 더 포괄적인 피드백 확보

### 4. **웹소설 특화 설계**
- 단순 소설이 아닌 "웹소설" 특성을 명확히 인식
- 회차 구성, 클리프행어, 후킹 전략 등 웹소설 특유의 요구사항 반영
- 한국 플랫폼(카카오페이지, 네이버 시리즈) 연재를 목표로 함
- **영향**: 실제 연재 가능한 수준의 품질 확보 가능

### 5. **확장 가능한 아키텍처**
- 장르별 에이전트 템플릿 구조로 새 장르 추가 용이
- 스킬 기반 설계로 공통 지식 재사용
- 커맨드/에이전트/스킬의 명확한 분리
- **영향**: 장기적으로 유지보수 및 확장이 쉬움

---

## ⚠️ 약점 및 개선 영역

### 1. **MVP 범위 과다**
- Phase 0만 해도 상당한 작업량 (3개 LLM 통합, 스킬, 에이전트, 커맨드)
- 3명 팀에서 Phase 0-4를 모두 구현하려면 상당한 시간 소요
- 빠른 MVP가 목표라면 범위가 지나치게 큼
- **영향**: MVP 출시 지연, 시장 기회 상실 위험

### 2. **외부 API 의존성 높음**
- Codex와 Gemini는 외부 유료 API 필요
- API 키 관리, 비용 발생, 가용성 이슈
- 개발 초기에는 추가 부담
- **영향**: 초기 비용 증가, 복잡도 상승

### 3. **에러 핸들링 및 폴백 전략 부족**
- scripts/call-codex.sh와 call-gemini.sh에 에러 처리 미흡
- API 호출 실패, 타임아웃, 응답 파싱 에러 대응 없음
- 사용자가 원인 파악 어려움
- **영향**: 개발 경험 저하, 디버깅 시간 증가

### 4. **테스트 전략 불명확**
- Phase 0에 "테스트" 항목만 있고 구체적 방법 부재
- 자연어 해석의 정확성을 어떻게 검증할 것인지 불명확
- 에이전트 품질 보증 방법 필요
- **영향**: 품질 리스크, 버그 발견 지연

### 5. **문체 가이드 미완성**
- writing-style 스킬이 "고민 중"으로 남아있음
- 공의 경계/룬의 아이들/하루키 스타일 적용 방식이 추상적
- 실제 소설 작성 시 일관성 확보 어려움
- **영향**: Phase 1 이후 진행 불확실성

---

## 💡 권장사항

### 🔥 Critical

#### 1. MVP 범위 대폭 축소
**현재 계획**: Phase 0-4 전체
**권장 MVP**:
- Phase 0 중 Claude만 먼저 구현 (Codex, Gemini 제외)
- `/review-claude` 커맨드 + plan-reviewer-claude 에이전트만
- review 스킬은 기본만
- **이유**: 3명 팀이 빠른 MVP를 목표로 할 때 외부 LLM 통합은 과도함
- **다음 단계**: Claude만으로 검증 후 필요시 Codex/Gemini 추가

#### 2. 에러 핸들링 및 사용자 피드백 강화
**스크립트에 추가**:
```bash
# API 호출 실패 처리
if ! RESPONSE=$(curl --fail -s ...); then
    echo "Error: API 호출 실패. 네트워크 또는 API 키를 확인하세요." >&2
    exit 1
fi

# 응답 검증
if ! echo "$RESPONSE" | jq -e '.choices[0].message.content' > /dev/null 2>&1; then
    echo "Error: API 응답 형식이 올바르지 않습니다." >&2
    echo "$RESPONSE" >&2
    exit 1
fi
```
- **영향**: 사용자가 문제를 즉시 파악하고 해결 가능

#### 3. API 키를 환경변수로 전환
**현재**: llm-config.json에 평문 저장
**권장**:
- `.env` 파일 사용 (`.gitignore`에 추가)
- 스크립트에서 `$OPENAI_API_KEY` 환경변수 읽기
- llm-config.json에는 모델명/파라미터만
- **이유**: 보안 강화, 실수로 커밋될 위험 제거
- **구현**:
```bash
API_KEY="${OPENAI_API_KEY:-$(jq -r '.openai.api_key' "$CONFIG_FILE")}"
```

---

### 📌 Important

#### 4. Phase 0 내에서도 우선순위 재조정
**현재**: 3개 LLM 모두 동시 구현
**권장 순서**:
1. Claude 에이전트 + 커맨드 (1-2일)
2. 기본 review 스킬 (1일)
3. 현재 계획 문서로 테스트 (0.5일)
4. 피드백 반영 후 Codex 추가 (선택, 1-2일)
5. Gemini 추가 (선택, 1-2일)

- **이유**: 점진적 구현으로 빠른 피드백 루프 확보

#### 5. 문체 가이드 간소화 또는 후순위
**현재**: 공의 경계/룬의 아이들/하루키 스타일 복잡하게 고려
**권장**:
- Phase 1에서 기본 웹소설 문체만 정의 (가독성, 간결함)
- 고급 스타일은 Phase 4로 후순위 이동
- **이유**: MVP에 집중, 복잡한 문체는 나중에 개선 가능

#### 6. 테스트 계획 구체화
**추가할 테스트**:
- 기본 요청: `/review-claude` (기본값 동작)
- 관점 지정: `/review-claude "엔지니어 관점으로"`
- 복합 요청: `/review-claude "staff+ 엔지니어 관점으로 plans를 리뷰해줘. 스타트업이야"`
- 에러 케이스: 파일 없음, 잘못된 요청
- **영향**: 자연어 해석 품질 보증

---

### 💭 Nice-to-have

#### 7. 리뷰 결과 캐싱
- 동일 파일에 대한 반복 리뷰 시 캐시 활용
- 파일 해시 기반으로 캐시 무효화
- **영향**: 비용 절감, 속도 향상
- **우선순위**: Phase 4 이후

#### 8. 리뷰 히스토리 추적
- `.review-output/`에 타임스탬프별 결과 보관
- 시간에 따른 계획 개선 추이 확인
- **영향**: 프로젝트 진화 과정 가시화
- **우선순위**: Phase 4 이후

## 🎯 종합 평가

**점수**: 82/100

**세부 점수**:
- 완성도: 19/20 (매우 상세하고 포괄적)
- 일관성: 18/20 (전체적으로 일관되나 문체 가이드 미완성)
- 명확성: 17/20 (명확하나 일부 추상적 부분 존재)
- 실현 가능성: 13/20 (3명 팀에 범위가 과도함)
- 품질: 15/20 (높은 품질이나 MVP 관점에서 과잉)

**최종 의견**:

탁월한 설계와 체계적인 접근 방식을 보여주는 계획입니다. 특히 자연어 인터페이스 채택과 다중 LLM 활용 전략은 혁신적이며, Phase 구분이 명확하여 실행 가능성이 높습니다.

하지만 **3명 스타트업 팀에서 빠른 MVP를 목표로 한다면 범위가 지나치게 큽니다**. Phase 0만 해도 상당한 작업량이며, 외부 LLM 통합은 초기 단계에서 과도한 복잡도를 추가합니다.

**핵심 권장사항**:
1. MVP 범위를 Phase 0의 Claude만으로 축소
2. 검증 후 점진적으로 Codex/Gemini 추가
3. 에러 핸들링 및 보안 강화
4. 문체 가이드 간소화

이렇게 조정하면 **1-2주 내에 실용적인 MVP를 출시**할 수 있으며, 이후 사용자 피드백을 받아 점진적으로 개선할 수 있습니다. 현재 계획은 "완벽한 시스템"을 목표로 하지만, 스타트업에서는 "작동하는 최소 시스템"을 먼저 만들고 빠르게 반복하는 것이 더 중요합니다.

## 예시 2: 소설 리뷰 (독자 관점)

### 요청
> "50대 웹소설 독자 입장에서 마탑의 귀환자 1화를 평가해줘"

### 리뷰 결과

**관점**: 50대 웹소설 독자 (일반 독자, 비전문가)
**대상 파일**: novels/마탑의귀환자/1화/소설.md

---

## 📋 요약

첫 장면부터 강렬한 인상을 줍니다. 주인공의 절박한 상황이 잘 전달되고, 과거와 현재를 오가는 구성이 흥미롭습니다. 다만 전문 용어가 다소 많아 이해하는 데 집중이 필요했고, 중반부 전개가 조금 느리게 느껴졌습니다. 마지막 클리프행어는 매우 효과적이어서 다음 화가 궁금해집니다.

---

## ✅ 강점

### 1. **강렬한 오프닝**
- 첫 문장부터 주인공의 죽음으로 시작하여 즉시 후킹됨
- "이번에는... 다르다"라는 대사가 신비감을 더함
- 독자의 호기심을 바로 자극
- **느낌**: 첫 화면부터 눈을 뗄 수 없었음

### 2. **공감 가는 주인공**
- 수많은 실패를 겪은 주인공의 절박함이 느껴짐
- "또 실패했다"라는 반복이 감정적으로 와닿음
- 나이 든 독자로서 인생의 재기 기회에 공감
- **느낌**: 주인공을 응원하게 됨

### 3. **명확한 목표**
- "마탑을 정복한다"는 목표가 명확
- 과거의 실수를 반복하지 않겠다는 결의
- 독자가 다음에 무슨 일이 일어날지 예측 가능
- **느낌**: 이야기의 방향을 알 수 있어 안심됨

### 4. **효과적인 클리프행어**
- 마지막에 "그 사람"이 등장하며 떡밥 투척
- "이번에는 다를 것"이라는 암시
- 다음 화가 매우 궁금해짐
- **느낌**: 바로 2화를 읽고 싶어짐

---

## ⚠️ 약점 및 개선 영역

### 1. **전문 용어가 많음**
- "마나 회로", "에테르", "시공 균열" 등 판타지 용어 과다
- 50대 독자로서 용어에 익숙하지 않아 이해에 시간 소요
- 설명 없이 등장하는 용어들이 몰입을 방해
- **영향**: 가끔 흐름이 끊김

### 2. **중반부 전개 느림**
- 회상 장면이 길어서 지루함
- 과거 실패 묘사가 반복적
- 조금 더 간결했으면 좋겠음
- **영향**: 중간에 집중력 떨어짐

### 3. **배경 설명 부족**
- "마탑"이 정확히 무엇인지 불명확
- 왜 주인공이 마탑을 정복해야 하는지 동기 약함
- 세계관이 애매함
- **영향**: 이야기에 완전히 몰입하기 어려움

---

## 💡 권장사항

### 🔥 Critical

#### 1. 전문 용어 설명 추가
- 처음 등장하는 용어는 간단히 설명
- 예: "마나 회로(마법사의 에너지 통로)"
- **효과**: 독자 이해도 향상, 진입장벽 낮춤

#### 2. 중반부 회상 축소
- 과거 실패 장면을 더 간결하게
- 핵심 감정만 남기고 세부사항 생략
- **효과**: 전개 속도 개선, 지루함 감소

---

### 📌 Important

#### 3. 마탑 설명 보강
- 마탑이 무엇인지, 왜 중요한지 초반에 언급
- 주인공의 동기를 더 명확히
- **효과**: 독자가 이야기에 더 몰입

**세부 느낌**:
- 흥미도: 8/10 (매우 흥미로움)
- 가독성: 7/10 (대체로 읽기 쉬우나 용어가 걸림돌)
