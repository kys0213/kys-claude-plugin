# 리뷰 가이드라인

## 리뷰 철학

### 5대 원칙

1. **건설적 (Constructive)**: 문제 지적 시 반드시 대안 제시
2. **구체적 (Specific)**: "코드가 복잡하다" → "함수 X의 cyclomatic complexity가 15로, 분리를 권장합니다"
3. **실행 가능 (Actionable)**: 바로 적용할 수 있는 제안
4. **균형 잡힌 (Balanced)**: 강점도 반드시 언급
5. **컨텍스트 인식 (Context-Aware)**: 프로젝트 상황, 팀 규모, 일정을 고려

### 리뷰어 태도

- **존중**: 코드 뒤에 사람이 있음을 기억
- **겸손**: "이것은 잘못되었다" → "이 방식도 고려해볼 수 있습니다"
- **명확함**: 모호한 표현 대신 구체적 제안
- **공감**: 제약 조건과 트레이드오프 이해

## 리뷰 프로세스

### Step 1: 문서 수집

리뷰 대상 파일을 수집합니다.

### Step 2: 관점 설정

요청된 관점에 맞게 평가 기준을 설정합니다.

### Step 3: 분석

각 평가 기준에 따라 문서를 분석합니다.

### Step 4: 피드백 작성

구조화된 피드백을 작성합니다.

### Step 5: 컨텍스트 반영

프로젝트 상황을 고려하여 최종 조정합니다.

## 기본 평가 기준

각 20점 만점, 총 100점:

| 기준 | 설명 | 배점 |
|------|------|------|
| 완성도 (Completeness) | 필요한 정보가 모두 있는가 | 20 |
| 일관성 (Consistency) | 내부적으로 모순이 없는가 | 20 |
| 명확성 (Clarity) | 이해하기 쉬운가 | 20 |
| 실현 가능성 (Feasibility) | 실제로 구현/실행 가능한가 | 20 |
| 품질 (Quality) | 깊이와 정교함 | 20 |

## 관점별 추가 기준

### 엔지니어 관점
- 기술적 건전성
- 확장성
- 유지보수성
- 기술 부채

### 보안 관점
- 인증/권한
- 입력 검증
- 데이터 보호
- OWASP Top 10

### 성능 관점
- 응답 시간
- 리소스 사용
- 병목 지점
- 캐싱 전략

## 점수 해석

| 점수 | 등급 | 해석 |
|------|------|------|
| 90-100 | Excellent | 바로 적용 가능 |
| 80-89 | Good | 경미한 수정 후 적용 |
| 70-79 | Acceptable | 일부 개선 필요 |
| 60-69 | Needs Work | 상당한 개선 필요 |
| 0-59 | Major Revision | 재작성 권장 |
